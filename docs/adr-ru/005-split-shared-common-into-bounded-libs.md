# ADR-005: Разделить общий `libs/common` на ограниченные библиотеки

- **Date**: 2026-05-09
- **Status**: Принято

---

## Контекст

На входе в task-03 в репозитории было четыре библиотеки в `libs/`:
`common`, `config`, `inventory`, `retail`. Из них `libs/common`
разросся в свалку — он содержал перечисления маршрутизации микросервисов
(`MicroserviceMessagePatternEnum`, `MicroserviceQueueEnum`,
`MicroserviceClientTokenEnum`, `AppNameEnum`), хелпер кеша Redis
(`CacheHelper`), middleware/декоратор/типы корреляционного ID, модули
клиентов RabbitMQ (`MicroserviceClientInventoryModule`,
`MicroserviceClientRetailModule`) и межсервисный интерфейс
`IOrderProductConfirm` (продублированный в `libs/retail/interfaces/`).

Раздутый `common` был проблемой для целевой формы ADR-004 в трёх
конкретных аспектах:

- Межсервисные контракты сообщений (перечисления + интерфейсы) были
  смешаны с инфраструктурой фреймворка (Nest middleware, модули
  `@nestjs/microservices`), так что любой модуль, которому нужно
  было перечисление маршрутизации, транзитивно тянул в себя всё
  транспортно-зависимое дерево.
- Клей TypeORM жил в `libs/config/typeorm-module.config.ts`, что
  означало, что `app.module.ts` каждого сервиса напрямую конструировал
  `new TypeormModuleConfig(entities)`. Не было места, куда положить
  общую базовую сущность, общий mapper-aware-репозиторий или
  единственный источник истины для `SnakeNamingStrategy`.
- Применение `architecture-lint`, поставленное в очередь на task-12
  (по ADR-004), нуждается в различных типах элементов библиотек для
  написания правил границ. Единственный элемент `lib-common` сжимает
  пространство правил; разделение на библиотеки, названные по
  назначению (`contracts`, `database`, `messaging`, `cache`,
  `observability`, `ddd`), даёт каждой свой тип.

Это ADR фиксирует решение разделить `common` и окружающие библиотеки
на ограниченные библиотеки, названные по назначению, а также
структурные выборы внутри этого разделения (компоновка папок,
стратегия ID на `BaseEntity`, политика shim для окна миграции).

Разделение выполняется поэтапно: task-03 (эта задача) высаживает
**фундаментальные** библиотеки, от которых зависит остальная часть
миграции (`contracts`, `database`, утончённый `common`). Task-04
высаживает **интеграционные** библиотеки (`messaging`, `cache`,
`observability`, `ddd`). Task-06 строит `auth` с нуля.

---

## Решение

### Новые библиотеки (эта задача)

- **`@retail-inventory-system/contracts`** — межсервисные контракты
  сообщений и DTO. Только plain TypeScript (декораторы `class-validator`
  и `@nestjs/swagger` на DTO допускаются, потому что они и есть
  контракт; никакой зависимости от инверсии управления Nest).
  Подобласти:
  - `microservices/` — `AppNameEnum`, `MicroserviceClientTokenEnum`,
    `MicroserviceMessagePatternEnum`, `MicroserviceQueueEnum`. Примитивы
    межсервисной маршрутизации, не принадлежащие ни одному
    ограниченному контексту.
  - `retail/` — DTO заказа, перечисления, интерфейсы (перенесены из
    `libs/retail/`), плюс канонический дом `IOrderProductConfirm`.
  - `inventory/` — DTO остатков товара, типы payload и константы
    (перенесены из `libs/inventory/`).

- **`@retail-inventory-system/database`** — база TypeORM. Экспортирует
  `BaseEntity`, `BaseTypeormRepository<TEntity, TDomain>`,
  `SnakeNamingStrategy` (ре-экспорт `typeorm-naming-strategies`) и
  `DatabaseModule.forRoot(entities)` /
  `DatabaseModule.forFeature(entities)`. Модули приложений вызывают
  `DatabaseModule.forRoot(entities)` вместо прямой конструкции
  `TypeormModuleConfig`.

### Утончённый `@retail-inventory-system/common`

Сокращён до framework-free-утилитных каркасов:

- `result.ts` — дискриминированный union `Result<T, E>` плюс хелперы
  `ok` / `err`.
- `exceptions/domain.exception.ts` — абстрактный `DomainException` (без
  зависимости от Nest `HttpException`).
- `pagination/page.types.ts` — `IPageRequest`, `IPage<T>`.
- `types/utility.types.ts` — `Maybe<T>`, `Nullable<T>`.

В течение одного релиза библиотека также по-прежнему содержит `cache/`,
`correlation/` и `modules/` (модули `MicroserviceClient*Module`). Они
перемещаются в `libs/cache`, `libs/observability` и `libs/messaging`
в task-04 соответственно. Старый `libs/common/index.ts` ре-экспортирует
символы, перемещённые в contracts, чтобы существующие пути импорта
продолжали компилироваться в окне миграции; эти ре-экспорты удаляются
в task-14.

### Стратегия ID `BaseEntity`

`BaseEntity` использует автоинкрементный целочисленный `id`
(`@PrimaryGeneratedColumn()`). Рассмотренные компромиссы:

- Пять существующих сегодня сущностей (`order`, `order_product`,
  `order_status`, `order_product_status`, `product`, `product_stock`,
  `storage`, `customer`) уже используют автоинкрементные целые
  первичные ключи, поэтому переход на UUID потребовал бы одноразовой
  миграции данных и сломал бы все ранее выданные ID.
- У автоинкремента есть известные недостатки — последовательные ID
  утекают кардинальность, предсказуемы для злоумышленников и
  неудобны для генерации на стороне клиента до вставки. Ни один из
  них сегодня не несущий: нет публичной поверхности ID (gateway
  ещё не аутентифицирован — task-06), нет шардинга и нет потока
  pre-insert ID.
- Переход на UUID v7 (упорядоченный по времени) — это
  ориентированный на будущее выбор, если проект примет
  мульти-тенантную или шардированную форму. Это будущее ADR, когда
  такая потребность материализуется.

`BaseEntity` также включает `createdAt`, `updatedAt` и nullable
`deletedAt` для поддержки soft-delete TypeORM (`@DeleteDateColumn`).
Существующие сущности не переоформляются для наследования от
`BaseEntity` в этой задаче; эта конвертация — часть task-08
(inventory align) и task-09 (retail align).

### Политика shim для окна миграции

Введены три класса shim, и все они удаляются в task-14:

1. **`libs/common/index.ts`** ре-экспортирует символы, перемещённые в
   contracts (`AppNameEnum`, `MicroserviceClientTokenEnum`,
   `MicroserviceMessagePatternEnum`, `MicroserviceQueueEnum`,
   `IOrderProductConfirm`).
2. **`libs/inventory/index.ts`** и **`libs/retail/index.ts`**
   сводятся к единственной строке
   `export * from '@retail-inventory-system/contracts'`. TS-пути для
   этих библиотек остаются в `tsconfig.json` на один релиз.
3. **`libs/config/typeorm-module.config.ts`** сохраняет класс
   `TypeormModuleConfig` без изменений (вариант B в task-03 шаге 7 —
   «kept as a shim»). Логика фабрики теперь принадлежит
   `DatabaseModule.forRoot()`; класс функционально эквивалентен и
   остаётся только чтобы не вынуждать миграцию всех downstream-
   потребителей в одной задаче одновременно со структурным
   перемещением.

Места импорта приложений перенаправлены на новые пути в этой задаче;
shims — это защита глубокого эшелонирования для любого потребителя
(или будущей задачи), который ещё не перенаправлен.

### Выбор компоновки

`libs/contracts` и `libs/database` — это плоские каталоги с `index.ts`
в корне, отражая существующую форму
`libs/{common,config,inventory,retail}`. У них **нет** подпапки `src/`,
своего `tsconfig.lib.json` или записи в `projects` `nest-cli.json`.
Четыре существующие библиотеки следуют этой же плоской форме (см.
базовый `workspaces.json`); соответствие новых библиотек этой форме
удерживает все шесть библиотек единообразными.

## Рассмотренные альтернативы

**Сохранить все перечисления и интерфейсы в `libs/common`.** Отклонено.
Перечисления маршрутизации микросервисов и интерфейс
`IOrderProductConfirm` — это межсервисные контракты; размещение их в
той же библиотеке, что содержит Nest middleware и модули клиентов
RabbitMQ, заставляет любого потребителя контракта втягивать в себя
транспортно-зависимый код фреймворка. Весь смысл `libs/contracts` в
том, что файл доменного слоя в
`apps/<service>/src/modules/<module>/domain/` может импортировать
контракт, не принимая зависимость от Nest. Целевая гексагональная
форма ADR-004 запрещает эту связанность на уровне линта (task-12),
поэтому разделение должно состояться до того, как правила линта
включатся.

**Превратить каждую библиотеку в реальный Yarn-workspace** (со
своим `package.json`). Отклонено. Четыре существующие библиотеки —
это только TS-path-алиасы — никакого `package.json`, никакой записи
в `yarn workspaces list`. Добавление реальных workspaces для новых
библиотек ввело бы гибридное состояние (некоторые библиотеки —
workspaces, некоторые — нет), вокруг которого пришлось бы обходиться
в task-12. Продвижение всех шести библиотек до workspaces правдоподобно,
но вне области этой задачи; оставляется на будущее ADR, если граф
сборки этого потребует.

**Сохранить `TypeormModuleConfig` как поддерживаемое API и пропустить
`DatabaseModule`.** Отклонено. Класс-фабрика работает для того
подключения, которое выполняет сегодня, но не может содержать
`BaseEntity`, общий базовый репозиторий или разделение `forFeature`,
которое будут использовать будущие гексагональные компоновки по
модулям (каждый модуль будет регистрировать свои сущности через
`DatabaseModule.forFeature`, а не агрегировать в `app.module.ts`).
Как только эта возможность понадобится, `DatabaseModule` всё равно
появится; постройка его в этой задаче даёт task-08 и task-09 то,
что потреблять.

**Переместить `correlation/`, `cache/` и модули
`MicroserviceClient*Module` в их финальные дома в этой задаче.**
Отклонено как расширение области. Эти три заботы нетривиальны —
`correlation/` получит обогащение trace-id для OTel (task-10),
`cache/` получит форму порт/адаптер и сегмент схемы с версией из
пунктов `AUDIT-2026-05-08`, а модули messaging получат точечные
константы ключей маршрутизации. Сделать всё это в task-03 запутало бы
перемещение фундамента с интеграционным перемещением. Этими
перемещениями владеет task-04; эта задача их только подготавливает.

## Последствия

### Положительные

- Гексагональная цель (ADR-004) получает чистый шов общих контрактов:
  доменные слои в любом модуле могут импортировать контракт, не
  принимая зависимости от фреймворка Nest.
- Архитектурный линт (task-12) получает различные типы элементов
  библиотек (`lib-contracts`, `lib-database`, `lib-common`,
  `lib-config`), против которых писать правила границ, без коллапса
  раздутого `common`.
- `BaseEntity` и `BaseTypeormRepository` дают задачам выравнивания
  по сервисам (task-08 inventory, task-09 retail) единственное место
  для подключения soft-delete, хуков mapper и (позже) аудит-колонок.
- `DatabaseModule.forRoot()` убирает конструкцию `TypeormModuleConfig`
  из каждого `app.module.ts`, упрощая подключение приложений и
  централизуя решение о `SnakeNamingStrategy`.

### Отрицательные / компромиссы

- Окно миграции с shims в трёх местах (`libs/common`, `libs/inventory`,
  `libs/retail` плюс shim `TypeormModuleConfig` в `libs/config`) — это
  in-flight-код, который нужно держать до task-14. Ре-экспорты shim
  чётко помечены комментариями, чтобы будущий читатель мог с одного
  взгляда сказать, что является migration scaffolding, а что —
  продакшен-API.
- Выбор автоинкрементных ID для `BaseEntity` вместо UUID v7
  сохраняет текущую форму, но фиксирует проект на целочисленных PK,
  пока мы не пересмотрим. Компромисс задокументирован выше; если
  будущее ADR примет UUID, `BaseEntity` — правильное место для смены
  стратегии.
- Плоская компоновка библиотек (без `src/`, без `tsconfig.lib.json`)
  означает, что новые библиотеки не зарегистрированы в `projects`
  `nest-cli.json`. Они нормально разрешаются через пути
  `tsconfig.json`, но Nest-схематики, осведомлённые о `nest-cli.json`,
  их не увидят. Если будущая задача стандартизирует проект на
  правильной форме Nest-библиотеки, миграция — это одна задача,
  затрагивающая все шесть библиотек разом.
