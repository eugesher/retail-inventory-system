# ADR-007: Структурированные логи Pino + корреляция трейсов OpenTelemetry

- **Date**: 2026-05-10
- **Status**: Принято

---

## Контекст

[ADR-001](001-structured-logging-with-pino.md) установил Pino как
структурированный логгер для каждого сервиса, с `correlationId`,
несомым в каждой строке лога и пробрасываемым через HTTP-заголовок
`x-correlation-id` / payload сообщения RabbitMQ. Это даёт нам
per-request-корреляцию между сервисами, но не структуру трейсов: мы
знаем, какие строки лога принадлежат одному запросу, но не знаем,
сколько занял downstream-вызов или где в графе вызовов возникла
ошибка.

Рекомендация миграции выделяет OpenTelemetry как следующий слой
наблюдаемости. Task-10 подключает фактический `NodeSDK`, экспортёр и
UI Jaeger; task-04 вводит **форму библиотеки**, чтобы `main.ts`
приложений могли уже объявлять импорт побочным эффектом, не ломаясь,
когда task-10 заполнит тело.

Это ADR документирует:

1. Шаблон корреляции трейс+лог, который мы принимаем.
2. Разделение обязанностей между этим ADR и ADR-001.
3. Что поставляется в task-04 против task-10.

## Решение

### `libs/observability` — дом для Pino и OTel

Совмещение Pino и OTel в одной библиотеке отражает то, что они решают
одну проблему с разных сторон — строки лога несут `correlationId`
(область запроса) **и** `traceId` / `spanId` (область графа вызовов),
и оба должны обогащаться в одной и той же точке жизненного цикла
запроса. Разделение на две библиотеки означало бы две точки входа и
два пути проброса trace-context.

### Импорт побочным эффектом для bootstrap OTel

`libs/observability/tracer.ts` импортируется как побочный эффект из
`main.ts` каждого сервиса **до** `NestFactory.create*()`:

```ts
import '@retail-inventory-system/observability/tracer';
import { NestFactory } from '@nestjs/core';
// …
```

Auto-instrumentation OTel (`@opentelemetry/auto-instrumentations-node`)
патчит модули HTTP, MySQL, Redis и AMQP во время require. Если
bootstrap Nest выполнится первым, патчинг произойдёт против модулей,
которые уже были инстанциированы, и спаны пропадут. Импорт побочным
эффектом — единственный способ гарантировать правильный порядок без
явного pre-init-скрипта.

В task-04 тело файла пустое. Task-10 заполняет конфигурацию SDK
(OTLP-экспортёр, пропагатор W3C trace-context, атрибуты ресурсов,
завязанные на `AppNameEnum`).

### Хук обогащения Pino для `traceId` / `spanId`

`LoggerModuleConfig.pinoHttp.hooks.logMethod` уже фильтрует шумные
Nest-контексты в dev. Тот же хук получает заглушку сегодня (no-op,
потому что `tracer.ts` пока не запускает SDK), которую task-10
подключит для чтения `trace.getActiveSpan()?.spanContext()` и слияния
`trace_id` / `span_id` в bindings.

Форма вывода после task-10 будет:

```json
{
  "level": 30,
  "time": 1762000000000,
  "app": "retail-microservice",
  "context": "OrderConfirmService",
  "correlationId": "abc-123",
  "trace_id": "1a2b3c…",
  "span_id": "4d5e6f…",
  "msg": "Order confirmed"
}
```

Двойное присутствие `correlationId` и `trace_id` намеренно:
`correlationId` — это то, что грепают люди (он задаётся человеком через
заголовок на тестовых запросах); `trace_id` — это то, что потребляют
Jaeger и OTel-экспортёр.

### `TraceContextInterceptor` для заголовков ответа

Nest-interceptor копирует контекст активного спана в per-request-
bindings Pino и (в task-10) в заголовок ответа (`traceparent`).
Interceptor поставляется в task-04 как passthrough, чтобы модули
приложений могли уже его регистрировать.

### Связь с ADR-001

ADR-001 сохраняет **Status: Accepted**. Это ADR **не** замещает его —
они покрывают взаимодополняющие заботы:

- ADR-001 владеет конфигурацией Pino (уровень, редактирование,
  транспорт, форматтер), middleware корреляционного ID и правилом
  межсервисного проброса для `correlationId`.
- ADR-007 владеет структурой трейсов, контрактом импорта побочным
  эффектом и хуком обогащения trace+log.

Если два когда-либо вступят в конфликт, ADR-001 побеждает по форме
логов; ADR-007 побеждает по подключению трейсов.

## Последствия

- **+** `main.ts` приложений могут принять импорт побочным эффектом
  в task-04 и забыть о нём. Task-10 высаживается без касания кода
  приложений.
- **+** Операторы получают один поток логов + один поток трейсов,
  которые можно объединить по `trace_id`. `correlationId` остаётся как
  человекочитаемый идентификатор.
- **−** Два корреляционных идентификатора на строку лога (один
  человеческий, один машинный). Смягчается документированием обоих в
  разделе observability README.
- **−** Импорты побочным эффектом легко сломать авто-форматтерами,
  которые переупорядочивают импорты. ADR отмечает порядок импортов
  как жёсткое требование; CI его сегодня не обеспечивает (правило
  линта запланировано на task-10).

## Рассмотренные альтернативы

- **Только OTel, отказаться от `correlationId`.** Отклонено: теряет
  человекочитаемость; тесты уже передают `x-correlation-id` для
  детерминизма.
- **Только Pino, бесконечно отложить OTel.** Отклонено: per-request-
  корреляция не даёт per-call-задержки, а ADR-004 обязывает к
  гексагональным границам, которые выигрывают от явной структуры
  спанов.
