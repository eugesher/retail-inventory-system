# ADR-003: Документирование архитектурных решений

- **Date**: 2026-05-08
- **Status**: Принято

---

## Контекст

В retail-inventory-system накопилось достаточно архитектурных решений —
Pino + корреляционные ID (ADR-001), Redis cache-aside для остатков
товара (ADR-002) и полная миграция на гексагональную архитектуру,
начатая примерно во время написания этого ADR, — чтобы *причина* каждого
выбора начала утекать в описания PR, сообщения коммитов и память
сопровождающих. Ни одно из этого не выживает в долгосрочной перспективе:
описания PR доступны для поиска, но неудобны; сообщения коммитов сжимают
решения в одно предложение; память ротируется с уходом людей из команды.
Будущие решения (гексагональная компоновка по сервисам, разделение
библиотек, введение JWT/RBAC, OpenTelemetry + Jaeger, архитектурные
правила eslint) только усугубят проблему.

Постоянный формат, который фиксирует каждое решение рядом с управляемым
им кодом, датированный и пронумерованный для перекрёстных ссылок,
снимает необходимость реконструировать причины из истории. Два
существующих ADR уже используют гибридную форму Нигарда/MADR (Статус,
Контекст, Решение, Рассмотренные альтернативы, Последствия); это ADR
кодифицирует эту форму и соглашение о трёхзначной нумерации с ведущими
нулями, чтобы будущие записи были единообразными без переговоров
индивидуальных авторов.

---

## Решение

Мы будем документировать значимые архитектурные решения в этом проекте
как ADR в каталоге [`docs/adr/`](.).

**Формат.** Каждое ADR использует существующую гибридную раскладку
Нигарда, установленную в [ADR-001](001-structured-logging-with-pino.md) и
[ADR-002](002-redis-cache-aside-product-stock.md):

- Заголовок: `# ADR-NNN: <решение в активной форме>`.
- Строки `**Date**` и `**Status**` сверху
  (Status — одно из: Proposed, Accepted, Superseded by ADR-NNN,
  Deprecated, Rejected).
- `## Context` — какую проблему решение отвечает и какие
  силы её формируют. Опишите ситуацию такой, какая она есть в этом
  кодовой базе, а не в абстракции.
- `## Decision` — что было решено, с достаточной конкретикой,
  чтобы будущий читатель мог распознать решение в коде без
  присутствия автора.
- `## Alternatives Considered` — хотя бы один отклонённый вариант,
  с обоснованием в один абзац. Этот блок и делает ADR
  защищаемым позднее: «мы рассмотрели X и отклонили его, потому что Y».
- `## Consequences` — как положительные, так и отрицательные/компромиссы.
  Честно по поводу затрат, которые накладывает решение.

**Нумерация.** ADR нумеруются **трёхзначными целыми с ведущими нулями**,
начиная с 001 (`001-structured-logging-with-pino.md`). На момент
написания следующий свободный номер — 004 (003 — это ADR). Номера
выделяются при первом коммите файла; не резервируйте номера заранее.
Если решение позже отменяется, напишите новое ADR, которое
**Supersedes** старое; не редактируйте старое ADR на месте, кроме как
переключив его `Status` и добавив однострочную ссылку.

**Слаг.** Имя файла —
`docs/adr/NNN-<короткий-kebab-case-слаг>.md`. Слаг резюмирует
решение, а не область; предпочитайте
`004-adopt-hexagonal-architecture-per-service.md` вместо
`004-architecture.md`.

**Когда писать.** Любое из:

- Выбор между двумя или более разумными альтернативами, в которых
  будущие участники могут засомневаться.
- Ограничение, наложенное на кодовую базу (например, «доменные слои
  не могут импортировать из `@nestjs/*`»).
- Отмена или замещение предыдущего ADR.

Багфикс — это не ADR. Рефакторинг, не меняющий границ,
не ADR. Добавление зависимости из эргономических соображений
не ADR. Добавление зависимости, которая фиксирует проект на классе
решений (например, смена ORM), **является** ADR.

**Индекс.** [`docs/adr/index.md`](.) добавляется в task-13 и
перечисляет каждый ADR с номером, названием, статусом, датой и
однострочным резюме. До тех пор листинг каталога служит индексом.

## Рассмотренные альтернативы

**Четырёхзначное заполнение (`0001-…`).** Шаблон MADR и многие
open-source-проекты используют четырёхзначное заполнение нулями. Отклонено
для этого проекта, потому что ADR 001 и 002 уже существуют в `main` с
трёхзначным заполнением; перенумерация их до миграции затронула бы
каждую существующую ссылку в README и аудит-документах ради
косметической выгоды.

**Строгий шаблон MADR (с таблицей `## Considered Options`,
драйверами решений и т. д.).** Отклонено как более тяжёлое, чем нужно
проекту на его текущем масштабе. Гибрид Нигарда в ADR-001/ADR-002 уже
содержит «Alternatives Considered» — самое важное поле MADR — и
добавляет «Consequences», которого MADR по умолчанию лишён. Подмешивание
полного шаблона MADR позднее — это неломающее изменение, если проект
дорастёт до него.

**Никаких ADR; полагаться на описания PR и сообщения коммитов.**
Отклонено по причине, изложенной в Контексте: описания PR доступны для
поиска, но неудобны, а сообщения коммитов сжимают решения в предложения,
которые теряют *причину* в течение месяцев.

## Последствия

### Положительные

- Новый участник может прочитать `docs/adr/` от начала до конца и понять
  форму системы без раскопок в git-истории.
- Архитектурный дрейф легче заметить: если PR противоречит принятому ADR,
  у рецензентов есть конкретная ссылка, на которую можно указать.
- Миграции, deprecations и отмены оставляют след. Поле Status ADR — это
  единственный источник истины о том, действует ли решение всё ещё.

### Отрицательные / компромиссы

- Поддержка ADR — на каждом участнике; написать одно — это несколько
  сотен слов работы, не дающей фич. Гибридный шаблон выше держит
  планку низкой (нет драйверов решений, нет матрицы оценок), но
  планка ненулевая.
- Трёхзначный потолок допускает до 999 ADR. Если мы когда-нибудь к нему
  подойдём, переключение на четырёхзначное заполнение потребует
  перенумерации — принимается как долгосрочная проблема, маловероятная к
  материализации.
- «Значимое» — оценочное суждение. Список под «Когда писать»
  сужает выбор, но не устраняет его; ожидаются
  периодические разногласия о том, заслуживает ли изменение ADR.
