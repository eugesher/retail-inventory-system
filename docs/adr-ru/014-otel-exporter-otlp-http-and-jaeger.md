# ADR-014: Экспорт OTLP/HTTP в локальный Jaeger через OpenTelemetry-коллектор

- **Date**: 2026-05-14
- **Status**: Принято

---

## Контекст

[ADR-007](007-pino-and-opentelemetry.md) обязал проект к OpenTelemetry
как слою распределённых трейсов и зафиксировал *форму* bootstrap —
импорт `libs/observability/tracer.ts` побочным эффектом как первая
строка `main.ts` каждого сервиса плюс хук `logMethod` в Pino,
обогащающий записи логов `traceId` / `spanId`. Task-04 поставил пустую
оболочку; task-10 заполняет тело.

Остаётся четыре конкретных вопроса:

1. Какой **протокол экспортёра** используют SDK для отправки спанов
   из Node-процессов?
2. Куда эти спаны попадают в локальной разработке?
3. Как продакшен подключается к вендорскому backend без изменения
   кода приложения?
4. Как полученный трейс распространяется через **четыре** сервиса,
   которые мы поставляем сегодня (gateway → retail → inventory →
   notification), учитывая, что три из четырёх хопов пересекают
   очередь RabbitMQ?

Это ADR фиксирует ответы.

## Решение

### Экспортёр: OTLP/HTTP, не OTLP/gRPC и не Jaeger-thrift

`libs/observability/tracer.ts` использует
`@opentelemetry/exporter-trace-otlp-http`. Endpoint читается из
`OTEL_EXPORTER_OTLP_ENDPOINT` и должен оканчиваться на `/v1/traces`.
Мы выбираем HTTP вместо gRPC по трём причинам:

- **Более простой boot-отпечаток.** OTLP/HTTP — это одна зависимость
  (`exporter-trace-otlp-http`); gRPC тянет `@grpc/grpc-js`, нативные
  шаги сборки и генерацию protobuf-кода. Ничего из этого не
  окупается в Node-сервисах, которые уже везде говорят HTTP.
- **Проще отлаживать.** Экспортёр шлёт JSON-over-HTTP на `:4318`.
  `curl -X POST` против коллектора воспроизводит то, что отправляет
  SDK; `tcpdump` против gRPC сложнее для случайной диагностики.
- **Идентичное конечное состояние.** Коллектор превращает OTLP/HTTP в
  что угодно downstream — Jaeger, Tempo, Honeycomb и т. д. Разница в
  задержке горячего пути между HTTP и gRPC пренебрежимо мала на
  наших объёмах.

Мы **не** используем экспортёр Jaeger-thrift. Этот формат legacy, и
сам Jaeger теперь принимает OTLP нативно.

### Локальный sink: Jaeger all-in-one за коллектором

Локальный стек — это два контейнера:

```
apps (host or compose) ──OTLP/HTTP──► otel-collector:4318 ──OTLP──► jaeger:4317
                                                                       │
                                                                       └─► Jaeger UI :16686
```

Коллектор **не** строго необходим для локальной разработки — Jaeger
all-in-one принимает OTLP напрямую. Мы оставляем его, потому что:

- Он отражает продакшен-топологию: приложения никогда не общаются с
  вендором напрямую; они общаются с sidecar-коллектором, который
  владеет батчингом, повторными попытками и upstream-протоколом.
  Пропуск коллектора локально означал бы два разных графа для отладки.
- Процессор `batch` в коллекторе сглаживает всплески нагрузки.
  Локальные тесты иногда выстреливают десятки спанов за несколько
  сотен миллисекунд; батчинг избегает per-span-HTTP-roundtrip.
- Экспортёр `debug` коллектора печатает каждый спан, который
  эмитируют приложения, в stdout, что — самый быстрый способ
  подтвердить, что SDK подключён.

Конфиг коллектора живёт в `infrastructure/otel-collector-config.yaml`
и представляет собой единственный пайплайн: OTLP-приёмник →
процессор `batch` → OTLP-экспортёр (`tls.insecure: true`) к
`jaeger:4317`, с тройником `debug`-экспортёра.

### Compose-оверлей, не основной `docker-compose.yml`

Jaeger и коллектор живут в отдельном
`docker-compose.observability.yml`. Поднимаются командой:

```bash
docker compose -f docker-compose.yml -f docker-compose.observability.yml up
```

Причины держать их вне дефолтного стека:

- Они **opt-in**: большинство local-dev-сессий в них не нуждаются, а
  один только образ Jaeger ~250 МБ.
- Они разделяют сеть `backend`, определённую основным compose-файлом,
  так что URL service-to-service (`otel-collector:4318`) работают
  без изменений, когда оверлей поднят.
- Тесты, которым нет дела до трейсинга, не платят стоимость boot.

### Продакшен: тот же SDK, другое назначение коллектора

В продакшене каждый сервис по-прежнему публикует OTLP/HTTP в
коллектор. Конфигурация коллектора меняется — его экспортёр шлёт к
вендору (Honeycomb / Tempo / Datadog) вместо Jaeger — а приложения
остаются неизменными: один env-override `OTEL_EXPORTER_OTLP_ENDPOINT`
переключает хост. Это каноническая OTel-топология, и это причина, по
которой ADR-007 выбрал OTLP-через-коллектор вместо прямых вендорских
SDK.

### Auto-instrumentations покрывают весь граф вызовов

`@opentelemetry/auto-instrumentations-node` патчит модули, к которым
обращается каждый сервис: `http`, `mysql2`, `redis`, `amqplib`,
`nestjs-core` и другие. Хуки amqplib конкретно инжектят
`traceparent` в свойства AMQP-сообщения при публикации и извлекают
его при потреблении — именно так трейс четырёх сервисов остаётся
одним трейсом через RabbitMQ.

`amqp-connection-manager` (в `package.json`) — это обёртка вокруг
`amqplib`; нижележащие каналы — реальные amqplib-объекты Channel,
поэтому instrumentation прозрачно их патчит. Ручной smoke-тест для
task-10 подтвердил: запрос `PUT /api/order/:id/confirm` производит
трейс со спанами всех четырёх сервисов, включая пары `publish` /
`process` для `retail_queue`, `inventory_queue` и
`notification_events`.

### Требуемые переменные окружения

| Переменная | По умолчанию | Заметки |
| --- | --- | --- |
| `OTEL_SERVICE_NAME` | (обязательно) | Различается per service; помечает каждый спан через `Resource` |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | (обязательно) | Endpoint OTLP/HTTP traces (`…/v1/traces`) |
| `OTEL_RESOURCE_ATTRIBUTES` | опционально | Свободный список k=v, сливаемый в `Resource` |
| `OTEL_SDK_DISABLED` | `false` | Установите `true`, чтобы коротко замкнуть bootstrap — используется в некоторых тестовых контекстах |

Joi обеспечивает первые две как обязательные на boot. Отсутствующие
значения fail-fast — они молча не теряют трейсы.

### Обогащение спанов — обязанность auto-instrumentation

Мы **не** поставляем ручной декоратор `@Span()` на слое use-case.
Auto-instrumentation производит спан для каждого Nest-контроллера,
каждого Nest-микросервисного обработчика, каждого TypeORM-запроса,
каждого Redis-вызова и каждой AMQP-публикации/обработки. Получаемое
дерево уже — та форма, которую мы хотим (HTTP gateway → AMQP publish
→ retail handler → AMQP publish → inventory handler → AMQP publish →
notification handler плюс DB / Redis-дети на каждом уровне). Если
когда-либо потребуется кастомный спан (например, чтобы обернуть
не-инструментированную библиотеку), use case может вызвать
`trace.getTracer('app').startActiveSpan(...)` — но это остаётся
исключением, а не правилом.

## Последствия

- **+** Одна команда поднимает трейсинг локально:
  `docker compose -f docker-compose.yml -f docker-compose.observability.yml up`.
- **+** Продакшен переключается изменением назначения коллектора, а
  не кода приложения.
- **+** Трейс четырёх сервисов течёт бесплатно через RabbitMQ;
  никакого явного кода извлечения контекста на границах адаптеров.
- **−** Коллектор добавляет один хоп в dev. Приемлемая стоимость за
  паритет топологии с продом и за видимый debug-вывод.
- **−** Два ключа атрибута `Resource` (`service.name` и
  `deployment.environment.name`) написаны вручную, а не через
  `OTEL_RESOURCE_ATTRIBUTES`. Тривиально рефакторится позже.
- **−** Auto-instrumentation может производить шумные спаны для
  тривиального middleware (`middleware - patched`). UI Jaeger
  выводит их на видное место. Мы принимаем шум сейчас; сэмплинг /
  фильтрация по атрибутам спана — follow-up.

## Рассмотренные альтернативы

- **Прямые вендорские SDK** (Datadog `dd-trace`, Honeycomb beelines).
  Отклонено: фиксирует кодовую базу на одном вендоре; отклоняется от
  вендорнейтрального обязательства ADR-007.
- **Экспортёр Jaeger-thrift, без коллектора.** Отклонено: legacy-
  формат; продакшен почти наверняка не будет отправлять на инстанс
  Jaeger.
- **OTLP/gRPC.** Отклонено по причинам выше. Можно переключиться,
  поменяв одну зависимость и одну env-переменную, если объём трейсов
  вырастет.
- **Никакого трейсинга, оставить только корреляционные ID.**
  Отклонено в ADR-007. Корреляционные ID отвечают на «какие логи
  принадлежат этому запросу»; спаны отвечают на «куда ушла задержка»
  и «где возникла ошибка» — оба вопроса всплывают в межсервисном
  потоке подтверждения, который теперь у нас есть.
