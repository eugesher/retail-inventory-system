# ADR-001: Структурированное логирование с Pino и корреляционными ID

**Status:** Принято

---

## Контекст

Система состоит из четырёх NestJS-сервисов, которые взаимодействуют асинхронно через RabbitMQ. Когда запрос завершается ошибкой или ведёт себя неожиданно, для диагностики причины приходится читать вывод логов нескольких процессов — API gateway, принявшего HTTP-запрос; retail-микросервиса, обработавшего его; и inventory-микросервиса, который мог быть вызван в свою очередь.

До этого изменения каждый сервис использовал встроенный NestJS `ConsoleLogger` через тонкую обёртку `SystemLogger`, которая подавляла шум загрузки фреймворка. Это давало неструктурированный текст без контекста запроса:

```
[Nest] LOG [OrderCreateService] Order created
[Nest] LOG [ProductStockOrderConfirmService] Stock reserved
```

Не было способа определить, какие строки лога относятся к одному запросу, или программно запрашивать, агрегировать и выставлять оповещения по данным логов.

---

## Решение

Принят [Pino](https://github.com/pinojs/pino) через `nestjs-pino` для структурированного JSON-логирования во всех сервисах.

**Проброс корреляционного ID** реализован следующим образом:

1. `CorrelationMiddleware` выполняется на каждом входящем HTTP-запросе в API gateway. Он читает заголовок запроса `x-correlation-id`, генерируя UUID v4, если заголовок отсутствует. ID возвращается в заголовке ответа.
2. API gateway привязывает ID к своему pino-логгеру через `logger.assign({ correlationId })`. Все последующие вызовы лога в области одного HTTP-запроса автоматически включают его благодаря интеграции pino с `AsyncLocalStorage`.
3. ID включается в каждое исходящее сообщение RabbitMQ в виде поля `correlationId`.
4. Микросервисы извлекают `correlationId` из payload и явно включают его в каждый вызов лога. Микросервисы не используют `logger.assign()`, поскольку работают вне HTTP-контекста, где инициализируется `AsyncLocalStorage`.

**Конфигурация логгера** централизована в `libs/config/logger/logger.config.ts` в виде класса `LoggerConfig`, реализующего интерфейс `Params` из `nestjs-pino`. Он:

- Помечает каждую строку лога полем `app`, идентифицирующим сервис-источник.
- Редактирует заголовки `Authorization`, `Cookie` и `Set-Cookie`.
- Подавляет шум загрузки фреймворка (`InstanceLoader`, `RouterExplorer` и т. д.) в непродакшен-окружениях.
- Включает `pino-pretty` в непродакшен-окружениях для удобочитаемого вывода.
- Учитывает переменную окружения `LOG_LEVEL`, со значением `debug` по умолчанию в разработке и `info` в продакшене.

## Рассмотренные альтернативы

**Winston** — большее распространение в экосистеме, но значительно медленнее Pino (Pino — самый быстрый JSON-логгер для Node.js по бенчмаркам). Интеграция Winston с NestJS менее идиоматична и требует больше шаблонного кода для получения структурированного вывода. Его система транспортов тяжелее минималистичного подхода Pino с воркер-тредами.

**Встроенный NestJS Logger (`ConsoleLogger`)** — никаких дополнительных зависимостей, но даёт неструктурированный текст. Нет понятия контекста запроса, нет структурированного вывода с полями и нет интеграции с системами агрегации логов. Не позволяет программно фильтровать или выставлять оповещения по конкретным полям лога.

**OpenTelemetry** — более комплексное решение для наблюдаемости, но значительная дополнительная сложность (SDK, коллектор, экспортёр трейсов, конфигурация сэмплера) для системы такого масштаба. Структурированное логирование Pino — это предпосылка для трейсинга OpenTelemetry, а не его замена; эти подходы дополняют друг друга.

## Последствия

**Положительные:**

- Каждая строка лога — машинно-разбираемый JSON, что позволяет агрегацию с помощью инструментов вроде Datadog, Grafana Loki или Elasticsearch.
- Любой запрос можно отследить по всем сервисам, отфильтровав по единому значению `correlationId`: `jq 'select(.correlationId == "...")'`.
- Уровень логирования можно менять в runtime через переменную окружения `LOG_LEVEL` без изменения кода.
- Чувствительные заголовки редактируются автоматически и не могут утечь в конвейеры агрегации логов.
- Опыт разработчика сохранён: `pino-pretty` форматирует вывод в цветные удобочитаемые строки в непродакшен-окружениях.

**Отрицательные:**

- Объём логов слегка увеличивается по сравнению с текстовым логированием из-за накладных расходов JSON-полей на строку.
- Микросервисы не могут использовать `logger.assign()`, потому что выполняются вне области HTTP-запроса. `correlationId` приходится передавать явно в каждом вызове лога — небольшое требование к дисциплине.
- `pino-pretty` — это dev-зависимость, которая не должна работать в продакшене (обеспечивается проверкой `NODE_ENV` в `LoggerConfig`).
