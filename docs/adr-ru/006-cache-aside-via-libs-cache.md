# ADR-006: Cache-aside через порт и адаптер `libs/cache`

- **Date**: 2026-05-10
- **Status**: Принято

---

## Контекст

[ADR-002](002-redis-cache-aside-product-stock.md) ввёл паттерн Redis
cache-aside для запросов остатков товара. Реализация жила в
`libs/common/cache/cache.helper.ts` (реестр ключей) и
`apps/inventory-microservice/.../product-stock-common-cache.service.ts`
(чтение/запись/инвалидация напрямую против `Cache` из
`@nestjs/cache-manager`). Доменный код таким образом зависел от
конкретного типа `Cache` из `@nestjs/cache-manager`, из-за чего
юнит-тесты стопали Nest-shaped-зависимость, а фасад inventory был
привязан к конкретному выбору backend.

ADR-004 обязывает кодовую базу к гексагональной архитектуре по
сервисам. Это значит, что доменный код должен зависеть от **портов**
(интерфейсов, которыми он владеет), а не от типов инфраструктуры.
Кеш — одна из трёх I/O-забот, выделенных в рекомендации как
нуждающихся в доме port-and-adapter (наряду с messaging и
observability).

Это ADR фиксирует форму, которую мы вводим в task-04, чтобы
разместить контракт cache-aside из ADR-002 за портом, и явное
решение **сохранить** контракт дословно, пока мы перестраиваем
абстракцию.

## Решение

Ввести `libs/cache` со следующими экспортами:

| Экспорт | Роль |
|--------|------|
| `ICachePort` | Интерфейс, от которого зависит код домена/фасада. Методы: `get<T>`, `set<T>`, `del`, `wrap<T>` (read-through). |
| `CACHE_PORT` | DI-токен (`Symbol('CachePort')`), привязывающий порт к адаптеру на уровне Nest-контейнера. |
| `RedisCacheAdapter` | `@Injectable()`-реализация `ICachePort` против существующей связки `@nestjs/cache-manager` + `@keyv/redis`. |
| `CacheModule` | Nest-модуль, импортирующий `NestCacheModule.registerAsync(cacheModuleConfig)` и привязывающий `CACHE_PORT → RedisCacheAdapter`. |
| `cacheModuleConfig` | Перенесено из `libs/config` — существующая фабрика, без изменений. |
| `CACHE_KEYS` | Реестр ключей со значениями-функциями. Шаблоны производят идентичные строки, что и предыдущий `CacheHelper.keys`, так что продакшен-записи кеша переживают деплой. |
| `CacheHelper` | Backwards-compat-shim, делегирующий в `CACHE_KEYS`. Хранится на один релиз; фасад inventory мигрирует с него в task-08. |
| `@Cacheable({ key, ttlMs })` | Декоратор метода, оборачивающий метод в `port.wrap(...)` для opt-in-кеширования read-through. Обобщённое применение — task-11. |

Метод `wrap` — это контракт cache-aside: доменный код вызывает
`cache.wrap(key, ttl, () => loadFromDb())` и никогда не видит ветви
промаха/попадания кеша. Адаптеры свободно могут добавлять джиттер,
префиксы версии схемы или защиту от stampede, не меняя места вызова.

### Связь с ADR-002

ADR-002 сохраняет **Status: Accepted**. Это ADR уточняет его абстракцию,
не меняя контракта:

- Поведение TTL (по умолчанию 300 с, перезапись на ключ через
  `CACHE_TTL_MS_PRODUCT_STOCK`) — без изменений.
- Префикс ключа кеша (`stock:<productId>:`) и sentinel `*` для
  нефильтрованного ключа — без изменений. Комментарии аудита
  (CACHE-009 — CACHE-012) сохранены дословно в
  `libs/cache/cache-keys.ts`.
- Инвалидация SCAN+UNLINK в фасаде inventory — без изменений в
  task-04. Task-08 мигрирует фасад на `ICachePort` и решает, должен
  ли `del` обрасти overload `delByPattern`, чтобы поглотить логику
  инвалидации.

### Что это ADR явно **не** решает

- 17 открытых пунктов аудита из `docs/audits/audit-2026-05-08.md`
  остаются открытыми. Task-11 их переоценивает — task-04 только
  перестраивает абстракцию.
- Второй адаптер (in-memory, multi-tier) — вне области. Порт
  спроектирован поддерживать его, но второй адаптер в task-04 не
  поставляется.

## Последствия

- **+** Доменный код можно тестировать с заглушкой `ICachePort` на
  основе `Map` вместо мока `@nestjs/cache-manager`.
- **+** Будущие перемещения (in-memory-адаптер для тестов, multi-tier
  с L1 в процессе + L2 Redis) не требуют изменений в домене.
- **+** Декораторное кеширование у потребителей становится
  механическим.
- **−** Два пути в один и тот же экземпляр Redis на один релиз
  (фасад через `CacheHelper` и адаптер через порт), пока task-08 не
  консолидирует. Приемлемо, потому что оба пути производят
  идентичные ключи.

## Рассмотренные альтернативы

- **Оставить `CacheHelper` как есть.** Отклонено: оставляет доменный
  код связанным с `@nestjs/cache-manager`, блокирует гексагональную
  цель ADR-004.
- **Инжектить `Cache` из `@nestjs/cache-manager` напрямую с
  сервисом-обёрткой.** Отклонено: обёртка всё равно была бы в
  прикладном слое, каждый потребитель пере-оборачивал бы её, и
  применению правил границ в task-12 нечего было бы grep-ать.
