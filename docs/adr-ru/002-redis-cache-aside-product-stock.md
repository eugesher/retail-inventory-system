# ADR-002: Использовать Redis cache-aside для запросов остатков товара

- **Date**: 2026-05-08
- **Status**: Принято

---

## Контекст

Inventory-микросервис предоставляет эндпоинт запроса остатков (`GET /api/product/:productId/stock`, обслуживаемый через RPC `INVENTORY_PRODUCT_STOCK_GET`). Остаток хранится не как строка текущего баланса, а как append-only-журнал `product_stock`: каждое резервирование, поступление или корректировка записывает новую строку с положительным или отрицательным `quantity` для пары `(productId, storageId)`. Получение текущего баланса поэтому требует агрегации `SUM(quantity) ... GROUP BY storageId` по всем строкам для товара.

Это работает корректно, но не масштабируется. Стоимость агрегации растёт линейно с числом строк журнала на товар. Шаблон доступа смещён в сторону чтений (каждый поток заказа обращается к остатку; многие потоки делают это более одного раза) и слаб на запись (строка добавляется только при подтверждении заказа или поступлении). Повторение одной и той же агрегации `GROUP BY` для каждого чтения тратит время БД на результаты, которые меняются нечасто.

Слой кеша — естественный выбор, но он должен корректно взаимодействовать с транзакцией подтверждения заказа: данные об остатках устаревают сразу же после коммита подтверждения, а отдача устаревших данных запросу проверки последующего заказа могла бы привести к превышению остатков (overselling).

---

## Решение

Принят паттерн **cache-aside (ленивая загрузка)** с Redis в качестве backing store.

Кеш живёт в общем фасаде `ProductStockCommonService` в inventory-микросервисе. Оба API — чтения (`ProductStockGetService`) и записи (`ProductStockOrderConfirmService`) — проходят через него, что и делает возможной согласованную инвалидацию.

**Путь чтения (`ProductStockCommonService.get`):**

1. Если вызов несёт `EntityManager`, принадлежащий вызывающему коду, или `ignoreCache: true`, кеш пропускается и чтение идёт напрямую из БД. (Чтение внутри открытой транзакции может видеть незакоммиченные строки; кеширование этих данных повредило бы общий кеш для других вызывающих.)
2. Иначе ключ запрашивается через `ProductStockCommonCacheService.get`. Попадание возвращает закешированный `ProductStockGetResponseDto`.
3. На промахе выполняется агрегация в БД через `ProductStockCommonGetService.execute`.
4. Результат записывается обратно через `ProductStockCommonCacheService.set` с TTL `CACHE_TTL_MS_PRODUCT_STOCK` (по умолчанию `60000` мс).

**Формат ключа кеша** (`CacheHelper.keys.productStock` в `libs/common/cache/cache.helper.ts`):

```
stock:<productId>:<sorted-storageIds-joined-by-comma>
stock:<productId>:*                  # when no storageIds filter is supplied
```

**TTL** настраивается двумя env-переменными: `CACHE_TTL_MS_DEFAULT` (глобальное значение по умолчанию для Cache-модуля, применяемое к любому `set()` без указания TTL) и `CACHE_TTL_MS_PRODUCT_STOCK` (используется явно при кешировании ответа на запрос остатков). Обе по умолчанию равны `60000` мс. TTL — это страховочная сеть; явная инвалидация — основной механизм актуальности.

**Инвалидация** запускается в `ProductStockOrderConfirmService` после коммита транзакции подтверждения заказа. Она работает по принципу fire-and-forget — RPC-ответ не блокируется работой с кешем. Реализация обращается через `@nestjs/cache-manager` к лежащему ниже клиенту `@keyv/redis` и выполняет `SCAN MATCH stock:<productId>:*` по каждому затронутому `productId` параллельно, а затем `UNLINK` для каждого найденного ключа. `UNLINK` предпочтительнее `DEL`, потому что освобождает память асинхронно на стороне Redis. Защитная резервная ветка использует `DEL` по имени ключа для не-Redis backends (например, in-memory store, подменённого для юнит-тестов).

Вызов инвалидации до коммита был отклонён: параллельный читатель мог бы перезаполнить кеш из незакоммиченного состояния, после чего транзакция могла бы откатиться, оставив кеш и БД расходящимися.

**Постепенная деградация:** каждая операция кеша (`get`, `set`, `invalidate`) обёрнута в `try/catch`. Ошибки логируются на уровне `warn` и проглатываются. Сбой `get` возвращает `undefined` (тот же контракт, что и промах), и фасад проваливается в БД. Сбой Redis ухудшает задержку, но никогда — корректность.

## Рассмотренные альтернативы

1. **NestJS `CacheInterceptor` (кеширование на уровне HTTP)** — отклонено. Он кеширует на границе HTTP-обработчика в API gateway, а не там, где реально живёт состояние остатков (в inventory-микросервисе). У него нет понятия о внутреннем событии записи, таком как подтверждённый заказ, поэтому нет чистого способа инвалидировать закешированный ответ при изменении лежащего ниже журнала. Он также полностью пропустил бы путь RPC (другие микросервисы, вызывающие `INVENTORY_PRODUCT_STOCK_GET` через RabbitMQ, никогда не касаются HTTP gateway).

2. **Write-through cache** — отклонено. Write-through предполагает, что значение кеша имеет ту же форму, что и записываемое, поэтому `set()` при записи держит кеш в синхронизации. Таблица `product_stock` append-only и хранит дельты по строкам; закешированное значение — это агрегация `SUM ... GROUP BY`. Новая строка журнала не позволяет вычислить новую агрегацию без повторного чтения БД, что лишает смысла write-through и добавляет стоимости на пути записи. Cache-aside с явной инвалидацией лучше подходит для append-only-журнала.

3. **In-memory-кеш (без Redis)** — отклонено. Per-process in-memory-кеш был бы самым быстрым для случая одного инстанса, но inventory-микросервис развёртывается так, чтобы масштабироваться горизонтально, и `INVENTORY_PRODUCT_STOCK_GET` потребляется из общей очереди RabbitMQ. Per-instance-кеши расходились бы — кеш инстанса A не видел бы инвалидацию инстанса B, что приводило бы к несогласованным чтениям между репликами. Централизация на Redis избегает этого класса ошибок ценой одного дополнительного сетевого хопа.

## Последствия

### Положительные

- **Меньшая задержка чтения при попаданиях в кеш.** Попадание — один round-trip к Redis; промах — один round-trip к Redis плюс `SUM`/`GROUP BY` по журналу. С ростом объёма журнала разрыв увеличивается.
- **Снижение нагрузки на БД на пути чтения.** Повторные чтения для одной и той же комбинации `(productId, storageIds)` обслуживаются из Redis в окне TTL, что существенно для любого потока, который запрашивает остаток более одного раза за жизненный цикл запроса.
- **Масштабируемость чтений.** Дорогая агрегация выполняется один раз на промах кеша, а не один раз на запрос, поэтому добавление новых реплик Inventory не умножает нагрузку на БД пропорционально.
- **Отсутствие регрессии корректности при сбое Redis.** Ошибки чтения, записи и инвалидации логируются и проглатываются; система всегда откатывается к БД.

### Отрицательные / компромиссы

- **Redis теперь является runtime-зависимостью для пути чтения inventory-микросервиса.** Ранее он был провижионирован, но не использовался. Операционно Redis должен мониториться и иметь правильное сайзинг; брауноут Redis молча увеличит нагрузку чтения на БД.
- **Задержка инвалидации добавлена на путь записи.** Каждое успешное подтверждение заказа запускает параллельный SCAN на каждый товар с последующим UNLINK. SCAN инкрементальный, и `COUNT` установлен в `100` за цикл, так что стоимость ограничена, но не нулевая. Смягчается запуском инвалидации как fire-and-forget после коммита.
- **Ограниченное окно устаревания при сбое Redis.** Если подтверждение закоммичено, но инвалидация не может достичь Redis, закешированные остатки для затронутых ключей `(productId, storageId)` остаются устаревшими до истечения TTL (по умолчанию 60 с). Приемлемо для текущего масштаба; пересмотреть при ужесточении бизнес-правил.
- **Гонка чтения/записи в cache-aside.** Между промахом кеша и последующим `cache.set` параллельный писатель может закоммитить и инвалидировать, после чего исходный читатель запишет уже устаревший результат из БД. Сегодня нет single-flight или version-stamping; устаревание ограничено TTL. Отслеживается как `AUDIT-2026-05-08 [CACHE-001]`.
- **Форма кеша типизирована только на этапе компиляции.** Ломающее изменение в `ProductStockGetResponseDto` десериализовало бы старые in-flight-записи без runtime-валидации в течение одного окна TTL. Будущее смягчение: сегмент с версией схемы в ключе (например, `stock:v2:<productId>:...`). Отслеживается как `AUDIT-2026-05-08 [CACHE-003]`.
