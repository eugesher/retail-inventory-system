# ADR-015: Строки логов Pino несут `traceId` / `spanId` OTel

- **Date**: 2026-05-14
- **Status**: Принято

---

## Контекст

[ADR-001](001-structured-logging-with-pino.md) стандартизировал
Pino-через-`nestjs-pino` с `correlationId` в каждой строке —
человекочитаемый идентификатор, переживающий межсервисные переходы,
потому что middleware gateway его пробрасывает.
[ADR-007](007-pino-and-opentelemetry.md) обязал к OpenTelemetry для
*структуры трейсов* (per-call-задержка, дерево parent/child спанов).
[ADR-014](014-otel-exporter-otlp-http-and-jaeger.md) подключил
фактические SDK + коллектор + Jaeger.

Как только OTel живёт, каждая строка лога Pino, эмитированная внутри
server-side-обработчика, имеет соответствующий *активный спан* — но
Pino этого не знает. Без дополнительного подключения логи несут
`correlationId`, а трейсы несут `traceId`, и две системы живут в
отдельных плоскостях. Вопрос оператора «какие логи принадлежат этому
трейсу?» требует либо (a) объединения по `correlationId` *и* поиска
трейса по его собственному ID отдельно, либо (b) подключения ID друг
к другу.

Это ADR фиксирует вариант (b): каждая строка лога Pino обогащается
`traceId` и `spanId` активного спана, чтобы запросы логов можно было
фильтровать по трейсу.

## Решение

### Обогащение живёт в `LoggerModuleConfig.pinoHttp.hooks.logMethod`

Хук уже существовал — он подавлял шумные framework-контексты в dev.
Мы расширяем его, чтобы:

1. Резолвить `trace.getActiveSpan()?.spanContext()` на каждом вызове
   лога.
2. Если контекст спана валиден (ненулевые `traceId` / `spanId`),
   слить `{ traceId, spanId }` в первый аргумент, когда это запись,
   или добавить впереди новую запись bindings, когда это plain-строка.
3. Если активного спана нет (например, логи, эмитированные во время
   boot, до того, как любой запрос вошёл в систему), пропустить
   вызов без изменений — никакого шума `traceId: undefined`.

Хук — правильный шов, потому что:

- Он выполняется **на каждом вызове лога**, а не на каждом запросе.
  Другие опции обогащения Pino (`customProps`, `mixin`) все
  срабатывают per-record, но `customProps` вычисляется один раз при
  старте и разделяется между вызовами. `logMethod` гарантирует, что
  вызов `getActiveSpan()` происходит в тот же момент, когда Pino
  собирается писать — а это и есть момент, когда async-контекст OTel
  тот, который нам нужен.
- Он естественно компонуется с существующей веткой «отбрасывать
  шумные framework-контексты» — обе работают на каждой записи.
- Это библиотечный код: каждый сервис наследует поведение из
  `LoggerModuleConfig` без подключения на стороне приложения.

### Именование полей: `traceId` и `spanId`, не `trace_id` / `span_id`

Пример payload в ADR-007 использует `trace_id` / `span_id` (snake_case,
соответствующий wire-формату OTel). Мы намеренно отклоняемся к
camelCase, потому что:

- Каждое другое поле в строке лога Pino в этом проекте — camelCase
  (`correlationId`, `userId`, `orderId`). Snake_case-пара читалась бы
  как чужеродное тело в `jq`-запросах.
- Auto-instrumentations, которые сами декорируют логи (например, когда
  позже будет добавлен `instrumentation-pino`), эмитируют `trace_id` /
  `span_id`. Сосуществование обеих форм в одной строке приемлемо —
  операторы могут грепать любую; OTel-aware-sinks понимают snake_case-
  пару без конфигурации.

Если проект позже выберет
`@opentelemetry/instrumentation-pino` (который авто-инжектит
`trace_id`/`span_id` без нашего хука), этот хук становится
избыточным и удаляется. Сегодня пакет auto-instrumentation не
установлен.

### `correlationId` остаётся — они отвечают на разные вопросы

Два идентификатора на строку лога намеренно:

| Поле | Источник | Время жизни | Использование |
| --- | --- | --- | --- |
| `correlationId` | middleware gateway или заголовок клиента | весь запрос, один или много хопов | человеческий greb, детерминированные тестовые утверждения |
| `traceId` | OTel-контекст (`trace.getActiveSpan()`) | один трейс, может включать множество correlationId в batch-потоках | объединение логов ↔ UI Jaeger |

Запрос технически может иметь один `correlationId` и один `traceId`,
отображающиеся 1:1 сегодня — но batched / scheduled-потоки в будущем
могут эмитировать множество трейсов под одним `correlationId`, и
наоборот, трейс, начатый в долго работающей фоновой задаче, не имеет
`correlationId` вовсе. Сохранение обоих расцепляет эти случаи без
насаждения редизайна.

### Никакого HTTP-заголовка ответа для `traceparent` сегодня

`TraceContextInterceptor` поставляется как placeholder. Auto-
instrumentation уже инжектит W3C-заголовок `traceparent` на исходящие
HTTP-ответы для маршрутов, которые она патчит. Добавление нашей
собственной мутации заголовка сверху рисковало бы затенить
OTel-эмитированное значение. Если когда-либо понадобится кастомная
форма, тело interceptor высадится в то время.

### Юнит-тест утверждает контракт

`libs/observability/spec/logger.module.spec.ts` конструирует реальный
`BasicTracerProvider`, регистрирует `AsyncLocalStorageContextManager`,
запускает спан через `context.with(...)` и утверждает, что вывод
хука несёт совпадающие `traceId` / `spanId`. Второй тест утверждает,
что хук — passthrough, когда нет активного спана. Это наименьший
behaviour-anchoring-тест, охраняющий контракт — что-либо более
вовлечённое — это интеграция SDK с Jaeger, что — smoke-тест в
carryover task-10.

## Последствия

- **+** Каждая строка лога, выполняющаяся внутри request-обработчика,
  тривиально объединяется с деревом трейса в Jaeger.
- **+** Логи, эмитированные во время boot (без спана), остаются
  чистыми — никакого шума `traceId: undefined`.
- **+** Никакого подключения на стороне приложения: каждый сервис
  наследует поведение из `LoggerModuleConfig`.
- **−** Два корреляционных идентификатора на строку лога. README и
  ADR-007 оба это документируют; операторы изучают различие один
  раз.
- **−** Именование полей отклоняется от OTel-дефолта (`trace_id` vs
  `traceId`). Задокументировано; обратимо, если sink предпочитает
  snake_case.
- **−** Хук вызывает OTel API на каждой эмиссии лога. Вызов `O(1)` и
  возвращает `undefined`, когда SDK не зарегистрирован, поэтому
  юнит-тесты, не bootstrap'ящие SDK, не платят реальной стоимости.

## Рассмотренные альтернативы

- **`@opentelemetry/instrumentation-pino`.** Авто-инжектит
  `trace_id` / `span_id`. Отклонено сегодня, потому что добавляет
  зависимость и тонко переопределяет конфигурацию хуков Pino;
  hand-rolled-хук чисто компонуется с существующим фильтром
  «отбрасывать шумные framework-контексты». Мы пересмотрим, если
  ручной хук когда-либо дрейфует.
- **OTel log signal (`@opentelemetry/sdk-logs`).** Маршрутизирует
  логи в pipeline OTel рядом с трейсами. Более крупное изменение,
  более сложная story для совместимости с log-агрегаторами и даёт
  мало, пока Pino + Jaeger уже покрывают запросы оператора.
- **Отказаться от `correlationId` после высадки OTel.** Отклонено:
  тесты утверждают `correlationId` для детерминизма (значение
  выбирает клиент), а операторы тренировались на нём с ADR-001.
