# ADR-004: Принять гексагональную архитектуру для каждого сервиса

- **Date**: 2026-05-09
- **Status**: Принято

---

## Контекст

Сегодня retail-inventory-system использует компоновку «толстых сервисов»:
каждый микросервис хранит свои фичи в `app/api/<feature>/` с
сервисами для каждого действия, которые напрямую инжектят
`Repository<X>` и `ClientProxy`. Шаблон последователен, и разделение
по действиям — это реальная сильная сторона (уже частично присутствует
на момент начала миграции архитектуры), но отсутствие инверсии
порт/адаптер означает, что:

- TypeORM привязан к поверхности сервиса — нет места, где можно
  было бы выполнить сервис против поддельного репозитория, поэтому
  юнит-тесты сегодня либо мокают репозиторий на каждый вызов, либо
  полностью обходят вопросы персистентности.
- `ClientProxy` к RabbitMQ имеет ту же форму, что inventory-сервис
  держит для retail, что просачивает транспортные заботы в
  бизнес-логику и делает межсервисный контракт неочевидным.
- Новые сквозные заботы (OpenTelemetry — task-10, обобщённый
  cache-aside — task-11, отправка уведомлений — task-07) не имеют
  очевидного шва для подключения, кроме как прямо внутри классов
  сервисов, что усугубило бы связанность.

Pre-migration-рекомендация, которая привела к этой работе,
идентифицировала гексагональную архитектуру (Ports & Adapters) как
целевой шаблон: это единственный TypeORM-совместимый шаблон с
проверенной звёздами референсной реализацией в экосистеме NestJS, он
позволяет проекту сохранить текущий стек (TypeORM/MySQL/RabbitMQ/Redis/Pino)
и предоставляет чистый шов для работы с кешем, OTel и уведомлениями,
последовавшей за структурной миграцией. Это ADR фиксирует *обязательство*
по этой цели. Применение архитектурного линтинга, предотвращающего
дрейф, зафиксировано в
[ADR-017](017-architecture-lint-via-eslint-boundaries.md).

---

## Решение

Проект будет перестроен так, чтобы **каждый микросервис следовал
гексагональной компоновке по модулям**.

**Ограниченные контексты.** Миграция рассматривает следующие четыре
единицы как свои верхнеуровневые структурные деления, отражая папку
`apps/`:

- **retail** (`apps/retail-microservice`) — заказы, клиенты.
- **inventory** (`apps/inventory-microservice`) — товары, остатки товара,
  склады.
- **notification** (`apps/notification-microservice`) — исходящие
  уведомления (строится с нуля в task-07; сегодня заглушка).
- **gateway** (`apps/api-gateway`) — HTTP-граница плюс аутентификация
  (строится с нуля в task-06).

Контексты retail и inventory — это ограниченные контексты в DDD-смысле;
notification — это прикладной сервис над внешним каналом; gateway — это
граница/агрегатор. Гексагональная компоновка применяется ко всем четырём
единообразно, даже если только первые два сегодня несут нетривиальную
доменную логику, чтобы правила размещения и границы линтинга были
одинаковыми везде.

**Компоновка по модулям.** Внутри каждого микросервиса каждый модуль
(`auth`, `order`, `product-stock`, `notification`, …) следует
четырёхслойному разделению:

```
modules/<module-name>/
├── domain/             # entities, value objects, domain services, ports (interfaces)
├── application/        # use-cases, application services — orchestrates the domain via ports
├── infrastructure/     # adapters: TypeORM repositories, RabbitMQ clients, Redis caches, HTTP clients
└── presentation/       # NestJS controllers (HTTP for gateway, @MessagePattern for microservices)
```

Подробные обязанности слоёв, допустимые зависимости и соглашения об
именовании кодифицированы в абзаце «Forbidden imports» из `CLAUDE.md` и
обеспечиваются [ADR-017](017-architecture-lint-via-eslint-boundaries.md).
Основные моменты:

- `domain/` не может импортировать ничего за пределами собственного
  модуля (никаких `@nestjs/*`, никакого TypeORM, никаких декораторов
  `class-validator` на сущностях).
- `application/` может импортировать из `domain/` и из инжектированных
  портов; не импортирует адаптеры инфраструктуры напрямую.
- `infrastructure/` реализует порты, объявленные в `domain/`, и
  содержит единственно разрешённые импорты TypeORM, RabbitMQ, Redis и т. д.
- `presentation/` содержит контроллеры, которые зависят от
  use-cases в `application/` через `@Inject(...)`; бизнес-логики здесь
  нет.
- Межмодульные импорты идут через контракты `@retail-inventory-system/<lib>`,
  никогда через глубокие пути в другой модуль.

**Именование.** Префикс `I*` для интерфейсов и суффикс `*Enum` для
перечислений (уже обеспечиваемый в `eslint.config.mjs`) сохраняются.
Use-cases оканчиваются на `*.use-case.ts`; порты называются `I<Aggregate>Port`
или `I<Aggregate>Repository` и живут в `domain/ports/`; адаптеры
живут в `infrastructure/<concern>/` и оканчиваются на `*.adapter.ts` или
`*.repository.ts`.

**Последовательность миграции.** Структурные перемещения не входят в
область этого ADR; они выполняются через task-03 (foundation libs),
task-04 (integration libs), task-05 (gateway align), task-06 (auth
fresh build), task-07 (notification fresh build), task-08 (inventory
align), task-09 (retail-orders align). Применение архитектурного линтинга
(правила `eslint-plugin-boundaries`) поставлено в очередь на task-12,
как только каждый сервис достигнет целевой формы; до тех пор правила
генерировали бы шум быстрее, чем миграция смогла бы его разрешить.

**Вне области этого ADR.** Выбор CQRS, доменных событий или event
sourcing откладывается. Гексагональная архитектура совместима со всеми
тремя, но не требует их; добавление их выборочно — это будущее ADR,
если возникнет потребность.

## Рассмотренные альтернативы

**Awesome Nest Boilerplate / Tony133 (плоская компоновка).** Отклонено,
потому что оба плоские и зафиксировали бы проект в форме «толстых
сервисов», от которой мы уходим. Ни один из них не предлагает места для
подключения работы с кешем, OTel и уведомлениями без повторного
введения той же связанности, которую мы пытаемся убрать.

**Domain-Driven Hexagon / Ultimate Backend.** Отклонено как
непропорциональное текущему масштабу проекта и как несущее
несовместимые слои персистентности (Mikro-ORM + CQRS event sourcing).
Гексагональное разделение по модулям здесь — это примерно 80/20 от этих
шаблонов: та же дисциплина границ, ничего из церемонии.

**Сохранить текущую компоновку `app/api/<feature>/` и добавить правила
линтинга без перестройки.** Отклонено, потому что правила линтинга не
могут изготовить порт — сервисы всё равно инжектили бы `Repository<X>`
напрямую, а межмодульный контракт всё равно протекал бы через
перечисления message-pattern в `libs/common`. Без структурного
перемещения швы, которые миграция пытается создать, просто не
существуют, чтобы их обеспечивать.

**Монорепозиторий по ограниченным контекстам с общей библиотекой `core`.**
Правдоподобная альтернатива, если бы проект был больше. Отклонено
сейчас, потому что существующее разделение на четыре приложения уже
правильная гранулярность, и уже имеющиеся библиотеки (`common`,
`config`, `inventory`, `retail`) покрывают потребности межприложенческого
контракта без отдельной библиотеки `core`. Раздел 7 `recommendation.md`
перечисляет эти библиотеки как «already correct, preserve as-is» с
перенесением (а не перезаписью).

## Последствия

### Положительные

- Use-cases становятся юнит-тестируемыми против fakes для каждого порта,
  убирая текущую зависимость от моков TypeORM для тестов уровня
  сервиса. Покрытие прикладного слоя может вырасти без подъёма
  настоящей базы данных.
- Слой `infrastructure/` — единственное место, где живут адаптеры,
  поэтому замена (например, MySQL → Postgres, RabbitMQ → Kafka, KeyV/Redis →
  другой store) — это локальное изменение, а не правка по всему проекту.
- Сквозные заботы из последующих фаз (обобщение cache-aside в task-11,
  спаны OTel в task-10, отправка уведомлений в task-07) подключаются на
  границе application/infrastructure, а не проходят через классы сервисов.
- Архитектурный линт (task-12) получает чёткую цель: типы элементов
  отображаются напрямую на globs `domain | application | infrastructure | presentation`.

### Отрицательные / компромиссы

- Количество папок на модуль примерно утраивается (один `*.service.ts`
  становится портом, use-case, адаптером и контроллером). Для модулей
  с тонкой логикой — notification сегодня, контроллеры-проксисы
  gateway — структура может ощущаться церемониальной. Принимается как
  цена единообразия, поскольку правила архитектурного линта требуют
  одинаковой формы везде.
- Миграция многошаговая (tasks 03–09) и неизбежно перекраивает
  поверхность дифа. Шаблон «carryover-файлов», документированный в
  `CLAUDE.md`, является мостом: каждый task-NN.md производит
  `_carryover-NN.md` для следующей сессии.
- Правила линта (task-12) будут ловить back-edges (например, `domain/`,
  импортирующий TypeORM) только после завершения структурных
  перемещений; промежуточные PR задач могут кратковременно нарушать
  целевую форму. Это приемлемо, потому что каждая задача рецензируется
  и мерджится до начала следующей.
- Отказ от внедрения CQRS или event sourcing сейчас означает будущее
  решение, если асимметрия чтения/записи вырастет; оставляется как
  будущее ADR, а не переинвестируется сегодня.
