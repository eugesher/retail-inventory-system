# ADR-010: JWT-аутентификация и RBAC в API gateway

- **Date**: 2026-05-10
- **Status**: Принято

---

## Контекст

До task-06 в кодовой базе **не было ни аутентификации, ни авторизации**
ни на одном уровне: `package.json` не содержал `@nestjs/jwt` /
`@nestjs/passport` / `passport*` / `argon2` / `bcrypt`, а папка `auth/`
под `apps/api-gateway/src/` не существовала. Каждый HTTP-маршрут был
достижим любым клиентом. Это приемлемо для базы аудита, с которой
началась миграция; для поставки — нет.

Task-06 добавляет сквозную аутентификацию и role-based-авторизацию.
Решения в этом ADR фиксируют *форму*, которую мы выбрали, а не
инкрементальные заметки задачи — те живут в файле `_carryover-06.md`
рядом со скриптом задачи и удаляются перед мерджем.

ADR-009 уже установил гексагональную компоновку по модулям для
gateway; auth — первый gateway-модуль с настоящим агрегатом `domain/`,
поэтому ADR также документирует, как этот агрегат вписывается.

## Решение

### 1. JWT (HS256) вместо session-cookies и внешнего OAuth

**Выбрано.** Stateless JWT (HS256) для access- и refresh-токенов,
подписанные двумя независимыми секретами (`JWT_ACCESS_SECRET`,
`JWT_REFRESH_SECRET`).

**Отклонено: server-side session-cookies.** Сессии вынуждают каждый
микросервис общаться с общим session store (сегодня Redis, но новый
SPOF) до авторизации любого запроса. JWT позволяют gateway передать
короткоживущий bearer-токен downstream-микросервисам, которые могут
валидировать его offline против того же public/secret-материала. Это
соответствует нашему существующему паттерну «никакого
кросс-сервисного общего состояния, кроме payload RabbitMQ» и удерживает
`libs/contracts/auth/` framework-free, чтобы будущие сервисы могли
валидировать без втягивания `@nestjs/passport`.

**Отклонено: вынос на внешний OAuth-провайдер (Auth0, Cognito,
Keycloak).** Миграция отдаёт приоритет проекту-портфолио без внешних
зависимостей; втягивание IdP затмило бы остальную работу в
операционном весе. Задокументировано как будущая опция — если будет
введено, агрегат `User` становится кешем профиля, а адаптер пароля
выводится из эксплуатации.

### 2. argon2id вместо bcrypt

**Выбрано.** `argon2` (npm) с `argon2id`, минимальные параметры
стоимости OWASP 2024 (`memoryCost: 19_456` KiB, `timeCost: 2`,
`parallelism: 1`), настраиваемые per-environment через
`AUTH_ARGON2_MEMORY_COST` / `_TIME_COST` / `_PARALLELISM`.

argon2id — рекомендуемый OWASP-дефолт для новых приложений:
memory-hard, GPU-resistant и победитель Password Hashing Competition.
`bcrypt` всё ещё приемлем, но ограничен 72-байтовыми входами и не
memory-hard. Поскольку это greenfield-поверхность аутентификации (нет
legacy-хешей для миграции), нет причин выбирать более старый примитив.

### 3. Ротация refresh-токена с обнаружением повторного использования

**Выбрано.** Каждый успешный `/auth/refresh`:

1. Проверяет подпись и срок действия refresh JWT.
2. argon2-сверяет представленный токен с хешем, сохранённым в строке
   `user` (`refresh_token_hash`).
3. При совпадении: выпускает новые access- и refresh-JWT, сохраняет
   новый хеш, возвращает оба. Старый refresh-токен теперь недействителен
   (его хеш больше не совпадает).
4. При несовпадении: очищает живой `refresh_token_hash` (защитная
   мера — предполагаем компрометацию) и возвращает 401. Это
   срабатывает как circuit-breaker против злоумышленника, укравшего
   refresh-токен: следующий refresh легитимного пользователя
   провалится, вынуждая свежий вход.

Хранится хеш (не сырой токен), так что утечка БД не передаёт
атакующим воспроизводимые refresh-токены. Argon2 верифицирует за
~5–10 мс на вызов при наших параметрах стоимости — нетривиально, но
приемлемо для операции, выполняемой раз в 15 минут.

**Отклонено: stateless refresh.** Без сохранённого хеша нет канала
отзыва: украденный refresh-токен действителен до истечения его
7-дневного срока жизни.

### 4. Агрегат `User` живёт в gateway

**Выбрано.** `apps/api-gateway/src/modules/auth/` владеет `User`.
Gateway получает соединение TypeORM
(`DatabaseModule.forRoot([UserEntity])`).

**Отклонено: выделенный `user-microservice`.** Добавляет новый
RabbitMQ-хоп на **каждый** аутентифицированный запрос — callback
`validate()` JWT-стратегии должен был бы RPC-вызывать для поиска
активного пользователя. Удар по задержке неприемлем для горячих
путей, а поверхность деплоя растёт без кратковременной выгоды.

**Отклонено: `User` живёт в `retail-microservice` рядом с
`customer`.** Строка `customer` в retail представляет покупателя, а
не аутентифицированного principal: будущий «store manager»-админ
может не иметь строки `customer` и всё равно нуждаться во входе.
Связывание двух сейчас вынудило бы рефакторинг позже.

### 5. Все маршруты защищены по умолчанию

**Выбрано.** `JwtAuthGuard` и `RolesGuard` зарегистрированы как
глобальные провайдеры `APP_GUARD` в
`apps/api-gateway/src/app/app.module.ts`. Каждый маршрут требует
bearer-токен, если только обработчик или контроллер не несёт
`@Public()` из `@retail-inventory-system/auth`.

Opt-out **явный на маршруте**, а не в конфигурационном списке: никто
не может случайно открыть новый endpoint, забыв добавить его в
общий массив «protected routes». Два `@Public()`-маршрута сегодня —
`POST /auth/login` и `POST /auth/refresh`; всё остальное (включая
`/auth/logout`, `/auth/me`, `/order/*`, `/product/*/stock`) требует
валидный access-токен.

`@Roles(RoleEnum.X, …)` наслаивает role-based-авторизацию сверху.
Существующие маршруты помечены `@Roles(RoleEnum.CUSTOMER, RoleEnum.ADMIN)`.
Роль admin наследует доступ customer, будучи засеянной с обеими
ролями — иерархии ролей в логике guard нет.

### 6. Верификация токена downstream-микросервисами отложена

`libs/contracts/auth/` экспортирует форму `IJwtAccessPayload`, чтобы
будущий микросервис мог валидировать токен offline. Сегодня
downstream-микросервисы доверяют gateway: payload несут
`correlationId`, но пока не JWT. Проброс bearer или урезанного
principal через payload RabbitMQ — **будущая задача** — она стоит
изменения формы `libs/contracts`, обновления порта-публикатора и
верификации на стороне `@MessagePattern`. Не в области task-06.

Когда эта работа высадится, она вписывается в wire-формат ADR-008,
не ломая существующих мест вызова: новое опциональное поле
`authContext` на payload, с заполнением на стороне gateway и
`JwtAuthGuard` на стороне микросервиса (смонтированным на RPC-
транспорт). Это ADR заранее обязуется к **HS256 с
`JWT_ACCESS_SECRET`, общим между сервисами** для кросс-сервисного
варианта — RS256 — будущая опция, как только в области появится
правильный механизм распределения ключей (HSM, Vault).

### 7. Публичная регистрация отложена

`RegisterUserUseCase` существует и юнит-тестирован, но **не**
экспонирован через HTTP. Seed-пользователи (`admin@example.com` /
`customer@example.com`) покрывают сегодня каждый тестовый сценарий;
живой поток регистрации нуждается в rate limiting, верификации email и
CAPTCHA, прежде чем его можно будет безопасно экспонировать, и ни
одно из этого не в области. Задокументировано как follow-up в
`_carryover-06.md`.

### 8. Smoke-endpoint для guard роли admin

`GET /auth/admin/ping` существует исключительно чтобы дать role-guard
admin-only-цель, чтобы путь 403 customer-vs-admin был проработан в
E2E-тестах. Это не продакшен-поверхность управления пользователями;
если реальный admin-API всплывёт в более поздней задаче, эта заглушка
будет заменена.

## Последствия

- Gateway приобретает соединение TypeORM; до task-06 это было только у
  микросервисов. С точки зрения миграции это значит, что `app.module.ts`
  gateway теперь вызывает `DatabaseModule.forRoot([UserEntity])`.
  Соединение разделяется с базой MySQL retail/inventory-микросервисов
  (тот же `DATABASE_URL`), что удерживает Docker довольным ценой
  кросс-сервисной связанности записи на таблице `user` — только
  gateway в неё пишет.
- `libs/auth/` — это Nest-aware-библиотека. Доменный код (под
  `apps/*/src/.../domain/`) не должен от неё зависеть; потребители —
  контроллеры/use-cases/модули gateway. Правило чистоты DDD из
  CLAUDE.md по-прежнему действует.
- Joi-схема конфига в `libs/config` теперь требует `JWT_ACCESS_SECRET`
  и `JWT_REFRESH_SECRET` (≥ 32 символа каждый, различные). Стек,
  запущенный без этих env-переменных, fail-fast на старте.
- Значение RoleEnum — `'admin' | 'customer'`. Добавление новой роли —
  это правка трёх файлов (`libs/contracts/auth/role.enum.ts`, seed,
  аннотация `@Roles(...)` маршрута) — достаточно мало, чтобы сегодня
  не требовалось дальнейшей косвенности.

## Рассмотренные альтернативы (сводка)

| Решение | Выбрано | Отклонено | Почему |
| -------- | ------ | -------- | --- |
| Тип токена | JWT (HS256) | Server-side-сессии; внешний OAuth | Stateless; нет SPOF; область портфолио. |
| Хеш пароля | argon2id | bcrypt | Рекомендация OWASP; greenfield. |
| Политика refresh | Ротация с обнаружением повторного использования | Stateless refresh | Отзыв; сигнал повторного использования. |
| Дом пользователя | Gateway | Новый микросервис; retail-микросервис | Задержка; соответствие концепции. |
| Политика маршрутов по умолчанию | Защищены | Opt-in-защита | Fail-closed на новых маршрутах. |
| Верификация токена downstream | Отложена | В области сегодня | Поставлено после введения порта-публикатора (task-08/09). |
