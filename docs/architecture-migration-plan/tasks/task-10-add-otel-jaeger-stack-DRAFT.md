## Goal

Make a single end-to-end trace visible in Jaeger that spans
gateway Ôćĺ retail Ôćĺ inventory Ôćĺ notification when a real
`createOrder` Ôćĺ `confirmOrder` flow runs. Pino log lines for that
flow include `traceId` and `spanId` so logs can be filtered by trace
in any log aggregator. The OTel SDK is configured via env, not code,
so production deployments can swap exporters without rebuilding.

## Steps

1. **Add `docker-compose.observability.yml`** at the project root with:
   - `jaeger` (image: `jaegertracing/all-in-one:latest`, ports
     `16686` UI, `4317`/`4318` OTLP gRPC/HTTP receivers).
   - `otel-collector` (image: `otel/opentelemetry-collector-contrib`)
     with a config file at
     `infrastructure/otel-collector-config.yaml` (pipeline:
     receive OTLP Ôćĺ batch Ôćĺ export to Jaeger).
   The compose file is **separate** from the main `docker-compose.yml`
   so dev sessions can opt in (`docker compose -f docker-compose.yml -f docker-compose.observability.yml up`).

2. **Add OTel deps** to the root `package.json`
   (verify in task-01 / task-04: some may already be installed):
   - `@opentelemetry/sdk-node`
   - `@opentelemetry/auto-instrumentations-node`
   - `@opentelemetry/exporter-trace-otlp-http`
   - `@opentelemetry/api`
   - (optional) `@opentelemetry/instrumentation-amqplib` for explicit
     AMQP instrumentation.

3. **Flesh out `libs/observability/tracer.ts`** to:
   - Read `OTEL_EXPORTER_OTLP_ENDPOINT`,
     `OTEL_SERVICE_NAME`, `OTEL_RESOURCE_ATTRIBUTES` from env.
   - Configure `NodeSDK` with `OTLPTraceExporter` and the
     `getNodeAutoInstrumentations` set.
   - Call `sdk.start()` synchronously at module load; register a
     `process.on('SIGTERM')` shutdown hook.

4. **Add `OTEL_EXPORTER_OTLP_ENDPOINT` and `OTEL_SERVICE_NAME` to
   every `.env.example`** (verify in task-01: locate every
   `.env.example` ÔÇö there may be one per app and one at root).

5. **Verify trace propagation across RabbitMQ.** OTel auto-
   instrumentation for amqplib injects `traceparent` into message
   properties; confirm that on the consumer side the inbound
   message's `traceparent` is read into the active context. If the
   project uses `amqp-connection-manager` (which is in
   `package.json`), check that auto-instrumentation reaches it; if
   not, write a small interceptor in `libs/observability` that
   reads `properties.headers.traceparent` and binds it via
   `context.with(propagation.extract(...))`.

6. **Trace-correlate Pino logs.** In `libs/observability/logger.module.ts`,
   register a Pino mixin (or hook) that calls
   `trace.getActiveSpan()?.spanContext()` on each log emission and
   injects `traceId` / `spanId` into the log object. Update the
   `redact` list if any new field needs masking. Add a unit test
   that asserts a log emitted inside an active span carries the
   correct `traceId`.

7. **Manual smoke test.** With the observability stack up, drive
   the order flow (`POST /orders`, `POST /orders/:id/confirm`) and
   verify in Jaeger UI (`http://localhost:16686`) that a single
   trace shows spans across all four services. Capture the trace
   ID and a screenshot or text dump of the span tree into the
   carryover.

8. **Document the workflow.** Add a short section to `README.md`
   explaining how to start the observability stack and where to
   find the Jaeger UI.
