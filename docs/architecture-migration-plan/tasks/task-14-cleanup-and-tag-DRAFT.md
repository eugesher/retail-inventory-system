## Steps

1. **Remove deprecation shims.** If task-03 or task-04 left a stub
   re-export in `libs/common/index.ts` to ease the import-rewrite,
   delete it now. Run `yarn lint` and `yarn build` after every
   removal.

2. **Remove dead code.** For each app:
   - `find apps/<app>/src -type d -empty -delete` — but inspect the
     output before letting it run; legitimately-empty folders may be
     placeholders awaiting future work. Prefer manual deletion with
     a one-line note in the carryover.
   - Search for unused `*.service.ts` left over from the old
     pre-hexagonal layout (the hexagonal version is `*.use-case.ts`).
     Delete or repurpose.
   - Search for DTOs that are no longer imported anywhere (`grep -r`
     by symbol name).

3. **Resolve every `(verify in task-01)` annotation that the
   reconciler couldn't close.** If a draft annotated something and
   nothing later resolved it, this task either resolves it or
   explicitly carries it as a known limitation in the README's
   "Known issues" / "Future work" block.

4. **README polish.**
   - Top-level diagram: a Mermaid (or text) diagram showing the four
     services + RabbitMQ + MySQL + Redis + Jaeger.
   - "Architecture" section: pointer to `docs/adr/index.md` and a
     one-paragraph summary of the per-module hexagonal layout.
   - "Development" section: every `yarn` command from `package.json`
     listed with a one-line description.
   - "Observability" section (added in task-10): verify it still
     describes the current state.
   - Remove any pre-migration references that no longer apply.

5. **CLAUDE.md polish.** Same audit as the README. CLAUDE.md should
   describe the codebase as it is **after** migration: no
   "Notification microservice is a stub", no "Redis is provisioned
   but unused", no `app/api/<feature>/providers/` pattern. Replace
   with the per-module hexagonal pattern and the canonical template
   path.

6. **ADR index update.** Verify `docs/adr/index.md` lists every ADR
   under `docs/adr/` and that no ADR references a number that
   doesn't exist.

7. **Propose the release tag.** In `_carryover-14.md`, suggest the
   tag (`v1.0.0-architecture` or whichever the human prefers), the
   suggested release notes (a summary of the migration), and the
   suggested merge strategy (squash, merge commit, rebase).
   **Do not** run `git tag`, `git push`, `git merge`, or `gh release
   create`.

8. **Final verification.** Run the full check suite one more time
   and capture the output.
