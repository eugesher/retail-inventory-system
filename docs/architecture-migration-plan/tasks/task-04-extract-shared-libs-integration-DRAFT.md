## Goal

Land the four (or five) integration libraries — each with a port (or
constants/interfaces only, where there is no behavioural seam) and at
least one adapter — so that subsequent app-alignment tasks (task-05 to
task-09) can wire ports without inventing them. No app modules under
`apps/*/src/` are rewritten in this task; only their imports flip from
ad-hoc usages to the new libs.

## Steps

1. **`libs/messaging/`** — wrap RabbitMQ.
   - `messaging.module.ts` — provides the configured
     `ClientProxyFactory.create({ transport: Transport.RMQ, … })` reading
     `RABBITMQ_URL` from config.
   - `exchanges.constants.ts`, `routing-keys.constants.ts` — define
     `EXCHANGES = { RETAIL: 'retail', INVENTORY: 'inventory', NOTIFICATION: 'notification' }`
     and routing-key constants like
     `ROUTING_KEYS.RETAIL_ORDER_CREATED = 'retail.order.created'`.
     Migrate from the existing string-pattern enums under
     `libs/common/enums/` (verify in task-01: see
     `RETAIL_ORDER_CREATE`/`INVENTORY_PRODUCT_STOCK_GET`-style names
     that CLAUDE.md mentions; either rename to
     `<service>.<aggregate>.<event>` per the recommendation, or keep
     the existing names and update the recommendation to match —
     decided in task-01).
   - `rabbitmq.client.factory.ts` — factory that builds a
     `ClientProxy` for a given exchange/queue.
   - `decorators/` — thin re-exports of `@MessagePattern` /
     `@EventPattern` if the recommendation introduces a wrapper; skip
     if the wrapper isn't strictly needed.

2. **`libs/cache/`** — Redis cache-aside.
   - `cache.port.ts` — interface `CachePort` with `get<T>(key)`,
     `set<T>(key, value, ttl)`, `del(key)`, `wrap<T>(key, ttl, fn)`.
   - `redis-cache.adapter.ts` — implements `CachePort` using the
     existing `@keyv/redis` + `cache-manager` setup
     (already in `package.json`). Migrate the existing `libs/common/cache`
     wiring (per ADR-002 — verify in task-01) into this port/adapter
     shape rather than rebuilding.
   - `decorators/cacheable.decorator.ts` — method decorator that
     resolves a `CachePort` from the DI container and wraps the
     decorated method with read-through caching keyed by the decorator
     argument template (`'ris:retail:product:{id}'`).
   - `cache-keys.ts` — central registry of cache-key templates.
   - `cache.module.ts` — Nest module binding `CachePort` symbol to
     `RedisCacheAdapter`.

3. **`libs/observability/`** — Pino + OTel + correlation.
   - `tracer.ts` — boots the OTel `NodeSDK` with auto-instrumentation
     for HTTP, MySQL (TypeORM), Redis, AMQP. **This file must be
     importable as a side-effect with no Nest involvement** because
     it has to run before `NestFactory.create*()`.
   - `logger.module.ts` — wraps `nestjs-pino` with the project's
     redact list and a `traceId`/`spanId` enrichment hook
     (uses `@opentelemetry/api`'s active-span lookup).
   - `trace-context.interceptor.ts` — Nest interceptor that copies
     OTel context into Pino bindings for the lifetime of the request.
   - `http-context.middleware.ts` — request-id middleware
     (migrate from existing `libs/common/correlation` — verify in
     task-01).
   - `metrics.module.ts` — placeholder, may be empty until task-10.

4. **`libs/ddd/`** — framework-free building blocks.
   - `entity.base.ts`, `aggregate-root.base.ts`,
     `value-object.base.ts`, `domain-event.base.ts`,
     `repository.port.ts`. No `@nestjs/*`, no TypeORM. Pattern from
     Sairyss/domain-driven-hexagon scaled down to TypeORM-native
     usage.

5. **`libs/auth/` (if in scope per `_carryover-01.md`)**.
   - `jwt.strategy.ts`, `roles.guard.ts`, `current-user.decorator.ts`,
     `public.decorator.ts`, `auth.module.ts`. **However**: the
     orchestrator could not find any existing JWT/passport code or
     dependencies — this lib may be entirely new construction rather
     than a relocation. Task-01 will have decided whether to defer
     this to a separate, post-migration task. If the auth lib is being
     built fresh here, this step adds the dev dependencies
     (`@nestjs/jwt`, `@nestjs/passport`, `passport`, `passport-jwt`)
     and writes them as new files — but does **not** wire them into any
     app yet (that's task-05 for the gateway).

6. **Update `tsconfig.json` aliases** to include every new lib under the
   project's chosen prefix.

7. **Update `nest-cli.json`** so each new lib appears in the
   `projects` block as a `library` type with the correct `tsConfigPath`.

8. **Replace ad-hoc imports under `apps/`** with the new libs:
   - Anywhere a microservice imports the existing
     `MicroserviceClientRetailModule` / `MicroserviceClientInventoryModule`
     from `libs/common/modules`, repoint at `libs/messaging`.
   - Anywhere correlation-ID utilities are imported from
     `libs/common/correlation`, repoint at `libs/observability`.
   - Anywhere the existing `libs/common/cache` is imported, repoint at
     `libs/cache`.
   The apps must still compile; logic is not refactored in this task.
