## Goal

Establish the three foundation libraries that the rest of the migration
depends on: `libs/contracts` (cross-service message and DTO contracts —
plain TypeScript only, no Nest decorators), `libs/database` (TypeORM base
entity, base repository, snake-naming strategy, `DatabaseModule`), and a
slimmed-down `libs/common` that holds **only** framework-free utilities
(types, Result, exceptions, pagination). Do not touch app code yet; only
the libs change. Existing import sites that consume the old `libs/common`
or `libs/retail`/`libs/inventory` are updated to the new locations in this
task — the apps must still compile.

## Steps

1. **Plan the split.** Read `libs/common/`, `libs/config/`,
   `libs/retail/`, `libs/inventory/` end-to-end. Produce a mapping table:
   for each existing exported symbol, write its new home (`libs/contracts/<service>/`,
   `libs/database/`, `libs/common/`, or `libs/observability/auth/cache/messaging`
   in task-04). Save the table to `_carryover-03.md`. Do **not** start
   moving code until the table is complete.

2. **Create `libs/contracts/`** as a Nest library
   (`yarn nest g library contracts` — verify in task-01: confirm the
   monorepo's `nest g` works with Yarn 4 PnP; if not, hand-create the
   library directory and add the entry to `nest-cli.json`). Move all
   existing cross-service DTOs, message-pattern enums, RabbitMQ
   correlation-ID utilities, and shared interfaces into
   `libs/contracts/src/<service>/`. Keep the file names but reorganize
   into per-service folders: `libs/contracts/src/retail/`,
   `libs/contracts/src/inventory/`, `libs/contracts/src/notification/`,
   `libs/contracts/src/auth/` (only if `_carryover-01.md` keeps `libs/auth`
   in scope — otherwise omit).

3. **Create `libs/database/`** as a Nest library. Move (or create):
   - `base.entity.ts` — abstract class with `id` (uuid),
     `createdAt`, `updatedAt`, `deletedAt` columns.
   - `base-typeorm.repository.ts` — generic
     `BaseTypeormRepository<TEntity, TDomain>` with `find`, `save`,
     `softDelete`, and a mapper hook.
   - `snake-naming.strategy.ts` — re-export or wrap
     `typeorm-naming-strategies` so every service uses the same naming.
   - `database.module.ts` — wraps `TypeOrmModule.forRootAsync()` reading
     `DATABASE_URL` from `@nestjs/config`, applying the snake-naming
     strategy, and exposing `forFeature()` to consumers.
   (verify in task-01: existing `libs/config` already provides
   `TypeormModuleConfiguration` — decide whether to keep that name and
   relocate, or rename to `DatabaseModule` per the recommendation.)

4. **Slim `libs/common/`** to framework-free utilities only. Keep:
   - `result.ts` (a `Result<T, E>` type if/when needed),
   - `exceptions/` (domain-error base classes only — no `HttpException`),
   - `pagination/` (page/size/cursor helpers),
   - `types/` (shared TypeScript types).
   Move everything else out: enums to `libs/contracts/<service>/enums/`,
   correlation-ID utilities to `libs/observability/` (task-04), modules
   that wrap Nest to wherever they belong (task-04 will sweep up
   messaging, cache, auth, observability).

5. **Update path aliases in `tsconfig.json`** to add
   `@retail-inventory-system/contracts` and
   `@retail-inventory-system/database` (or `@app/...` per the
   `_carryover-01.md` decision). Do not remove existing aliases yet —
   the apps still depend on them.

6. **Update every import site under `apps/`** that referenced the moved
   symbols. The apps must still compile. If the recommendation chose to
   keep a stub re-export from `libs/common` for one release, leave
   re-exports in `libs/common/index.ts` so existing import paths keep
   working; otherwise rewrite the imports.

7. **Update each microservice's `*.module.ts`** so anything that previously
   imported `TypeormModuleConfiguration` from `libs/config` now imports
   `DatabaseModule` from `libs/database`.

8. **Verify** the apps still build and unit tests still pass.
