## Goal

Encode the module boundary rules from
`parts/recommendation.md` Section 3 as ESLint rules and enforce them
in CI. Every layer's allow-list/deny-list is expressed as
`boundaries/element-types`. The CI job fails the build on any
violation. Existing violations are fixed in this task; if any cannot
be fixed within scope, the task documents them as `--fix-later`
exceptions in `_carryover-12.md` and explicitly opens the discussion
for task-14.

## Steps

1. **Define element types** in `eslint.config.mjs`:
   - `domain` — files under `apps/*/src/modules/*/domain/**`.
   - `application-use-case` — files under
     `apps/*/src/modules/*/application/use-cases/**`.
   - `application-port` — files under
     `apps/*/src/modules/*/application/ports/**`.
   - `infrastructure` — files under
     `apps/*/src/modules/*/infrastructure/**`.
   - `presentation` — files under
     `apps/*/src/modules/*/presentation/**`.
   - `lib-contracts`, `lib-database`, `lib-messaging`, `lib-cache`,
     `lib-observability`, `lib-ddd`, `lib-common` — one element type
     per `libs/<name>`.
   - `app-bootstrap` — `apps/*/src/main.ts` and `app.module.ts`.

2. **Define rules** matching Section 3 of the recommendation:
   - `domain/` may only import `lib-ddd`, `lib-common`. Anything else
     errors.
   - `application-use-case` may import `domain`, `application-port`,
     `lib-ddd`, `lib-common`, `lib-contracts` (for command/query/view
     types only). Disallow `infrastructure` and any `lib-messaging`,
     `lib-database`, `lib-cache` import.
   - `application-port` may import `domain` types only.
   - `infrastructure` may import everything (this is where adapters
     live).
   - `presentation` may import `application` and `lib-contracts`,
     `lib-auth`. Disallow direct TypeORM `Repository` and direct
     Redis client imports.
   - `lib-contracts` is plain TypeScript (no `@nestjs/*`, no
     `typeorm`, no `class-validator` if you want to be strict —
     verify in task-01).
   - `lib-ddd` is framework-free (no `@nestjs/*`, no `typeorm`).
   - **Cross-service rule:** a file under `apps/X/...` may not
     import from `apps/Y/...` for any X ≠ Y.

3. **Run `yarn lint`** and fix every violation. Most should be
   trivial (an import that should have moved through a port).
   Anything genuinely hard goes in the carryover with a rationale —
   do not weaken the rule to make it pass.

4. **Replace the no-op `lint-architecture.yml` body** with a real
   job that runs `yarn lint` (or a dedicated script
   `yarn lint:architecture` that runs the boundaries rules only,
   for clearer CI failures). The existing main `ci.yml` lint step
   should already cover it; the dedicated job is for visibility and
   for the option to run boundaries-only on PRs.

5. **Add a regression test for the rules.** Write a lint-fixture
   file under `tests/lint/` that intentionally violates each rule
   and assert ESLint reports the expected error. This guards
   against silent rule weakening.
