## Goal

Capture an immutable baseline of the pre-migration repository (config
files, coverage report, dependency graph), install but **do not enable**
`eslint-plugin-boundaries` so the architecture-lint job has the dependency
ready when later tasks switch it on, and write a Phase-0 ADR recording the
migration as an architectural commitment. This task does not move any code.

## Steps

1. **Create `docs/baseline/`** and copy in (verbatim, untouched):
   - `nest-cli.json`
   - `tsconfig.json`, `tsconfig.build.json`
   - `package.json` (the dependency block + scripts; lockfile reference
     only — do not duplicate the lockfile)
   - `docker-compose.yml`
   - `eslint.config.mjs`
   - `jest.unit.config.js`, `jest.e2e.config.js`
   - `webpack.config.js`
   Each file gets a header comment at the top of the copy noting the date
   and commit SHA at which the baseline was captured. (Do **not** modify
   the originals.)

2. **Capture the test coverage baseline.** Run
   `yarn test:unit --coverage` and save the resulting `coverage/lcov-report`
   summary or `coverage/coverage-summary.json` to
   `docs/baseline/coverage.txt`. If unit tests don't currently produce
   coverage output, configure Jest just enough to do so for this baseline
   capture (`--collectCoverage --coverageReporters=text-summary`) — do
   **not** commit Jest config changes if the run is one-off; just record
   the output.

3. **Capture a dependency-graph snapshot.** Run
   `yarn workspaces list --json > docs/baseline/workspaces.json` (Yarn 4
   syntax) so later phases can reason about the workspace shape that was
   in place at the start.

4. **Install `eslint-plugin-boundaries`** as a dev dependency:
   `yarn add -D eslint-plugin-boundaries`. Do **not** add any rules to
   `eslint.config.mjs` yet. Do **not** add the plugin to the `plugins`
   array yet. Just install the package so the lockfile carries it.
   (verify in task-01: confirm this is the right plugin choice; the
   alternative is `eslint-plugin-import` with `no-restricted-paths`,
   which is already partially configured if `eslint.config.mjs` uses
   `import` — verify before installing.)

5. **Add ADR-NNN "Adopt hexagonal architecture per service"** in
   `docs/adr/`. The ADR records the decision to migrate; the body cites
   `parts/recommendation.md` as the target spec and lists the bounded
   contexts (retail, inventory, notification, gateway) plus the
   per-module `domain/application/infrastructure/presentation` layout.
   Number with the padding convention from `_carryover-01.md`.

6. **Add a CI no-op job** `architecture-lint` in
   `.github/workflows/lint-architecture.yml` that runs
   `yarn lint` (or eventually a dedicated lint config). For now the job
   does the same thing as the existing CI lint step — its purpose is to
   register the workflow file. Later tasks will switch it to a stricter
   architecture-only ruleset. (verify in task-01: existing CI shape may
   already cover this — check `.github/workflows/`.)
