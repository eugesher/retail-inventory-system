## Issues

### CACHE-001 — Cache-aside read/write race window

- **Code**: `CACHE-001`
- **Classification**: architecture
- **Description**: Classic cache-aside race: a reader reads a stale DB row (pre-commit), a writer commits + invalidates, the reader then writes the stale value to the cache. The window is small but real, especially under TTL=60s. Reproduction sketch: concurrent confirm + GET on the same `productId` with overlap between the GET's DB query and the writer's commit. Out of typical fix scope (needs single-flight or set-with-version). Related architectural concern: no stampede protection / single-flight on cache miss — concurrent first-misses on the same key fan out to N parallel DB queries (`p-limit` per key, `redis-stampede`, or a store-side advisory lock would address it). The two should likely be addressed together.
- **Referenced locations**:
  - `apps/inventory-microservice/src/app/common/modules/product-stock-common/product-stock-common.service.ts` (line ~73, get-miss path before `cache.set`)
- **Status**: unresolved

### CACHE-002 — Post-commit-only invalidate contract is comment-enforced

- **Code**: `CACHE-002`
- **Classification**: architecture
- **Description**: `invalidate` is exposed publicly with the post-commit-only contract enforced by comment, not by code. Easy to misuse — calling it inside a transaction would race with concurrent readers. Code-search reveals only one caller today (`product-stock-order-confirm.service.ts`). Future authors lack a compiler safeguard. A transaction-aware `addAndInvalidate` helper or an `afterCommit` callback registry would replace comment-enforced ordering with type-system-enforced ordering.
- **Referenced locations**:
  - `apps/inventory-microservice/src/app/common/modules/product-stock-common/product-stock-common.service.ts` (line ~102, `invalidate` method)
- **Status**: unresolved

### CACHE-003 — Untyped cache values; no schema-version segment in keys

- **Code**: `CACHE-003`
- **Classification**: architecture
- **Description**: `cache.get<ProductStockGetResponseDto>(...)` is a type assertion only; no runtime schema validation on read. A code change that alters the DTO shape leaves stale cached entries that read correctly on the JS side but with mismatched fields. Mitigation: include a schema-version segment in the cache key (e.g. `stock:v2:<productId>:...`) and bump it on breaking shape changes.
- **Referenced locations**:
  - `libs/common/cache/cache.helper.ts` (line ~1, file header above `CacheHelper`)
- **Status**: unresolved

### CACHE-004 — TTL has no jitter (thundering-herd risk)

- **Code**: `CACHE-004`
- **Classification**: architecture
- **Description**: TTL has no jitter — all entries written together expire on the same wall-clock band. Risk of thundering herd at the TTL boundary if traffic is correlated. Apply ±10% jitter on `cache.set`. Note: this interacts with the future stampede-protection fix in `CACHE-001` and may be bundled with it.
- **Referenced locations**:
  - `apps/inventory-microservice/src/app/common/modules/product-stock-common/providers/product-stock-common-cache.service.ts` (line ~28, class header)
- **Status**: unresolved

### CACHE-005 — Redis-down produces duplicate warn logs (get + set)

- **Code**: `CACHE-005`
- **Classification**: architecture
- **Description**: When `cache.get` throws, the method returns `undefined` (implicit). The façade then proceeds to a DB read **and writes the DB result back to the cache** in the next step. If the cache is fully unavailable the `set` will also throw and be swallowed (warn-logged). Net behavior is correct, but flow is implicit and produces duplicate warn lines per request during an outage. Reproduction sketch: kill Redis mid-test → GET still returns 200 with DB data; logs show two warns (read + write). Mitigation: add an explicit `cacheAvailable` flag from `get` to short-circuit the `set` attempt when the read already failed.
- **Referenced locations**:
  - `apps/inventory-microservice/src/app/common/modules/product-stock-common/providers/product-stock-common-cache.service.ts` (line ~32, class header)
- **Status**: unresolved

### CACHE-009 — Cache keys lack a tenant segment (latent multi-tenant collision)

- **Code**: `CACHE-009`
- **Classification**: bug
- **Description**: Cache keys are namespaced only by `productId`. There is no tenant/store segment, so once a multi-tenant model is introduced two tenants holding the same `productId` will read each other's cached stock — a correctness and data-leak bug. Mitigation: prepend the tenant identifier (e.g. `t:<tenantId>:stock:<productId>:`). No tenant model exists today, so this is latent.
- **Referenced locations**:
  - `libs/common/cache/cache.helper.ts` (line ~17, `keyPrefixes.productStock`)
- **Status**: unresolved

### CACHE-010 — `storageIds` sort comparator uses `charCodeAt(0)` only

- **Code**: `CACHE-010`
- **Classification**: bug
- **Description**: The `storageIds` sort comparator uses only the first character's char code (`a.charCodeAt(0) - b.charCodeAt(0)`). For any pair of storage IDs sharing a first character (e.g. `["ab", "aa"]`) the comparator returns 0, so the resulting order depends on the input order — and the cache key ends up input-order-dependent. Reproduction sketch: `GET ?storageIds=["ab","aa"]` then `GET ?storageIds=["aa","ab"]` produce two distinct cache keys, both miss-then-set. The consequence is extra cache misses (not wrong data). Proposed fix: use a full lexicographic compare via `a.localeCompare(b)`.
- **Referenced locations**:
  - `libs/common/cache/cache.helper.ts` (line ~37, sort comparator inside `keys.productStock`)
- **Status**: unresolved

### CACHE-011 — Literal `*` "all-storages" sentinel could be confused with a glob

- **Code**: `CACHE-011`
- **Classification**: bug
- **Description**: The "all-storages" cache key uses the literal `*` (`stock:<id>:*`), which mimics a glob pattern. SCAN MATCH for `stock:<id>:*` correctly matches the literal-`*` key plus per-storage keys today, but a future refactor that issues `cache.del('stock:<id>:*')` would silently miss it. Mitigation: use a non-meta sentinel such as `__all__`.
- **Referenced locations**:
  - `libs/common/cache/cache.helper.ts` (line ~38, `: '*'` literal in `keys.productStock`)
- **Status**: unresolved

### CACHE-012 — `invalidateNamedKeys` fallback covers only single-storage keys

- **Code**: `CACHE-012`
- **Classification**: bug
- **Description**: The fallback path for non-Redis backends only deletes the literal-`*` key and the single-`storageId` key per item. Multi-storage combo keys (e.g. `stock:1:storage-a,storage-b`) survive until TTL. Reproduction sketch: use an in-memory store + GET with `storageIds=["a","b"]` to cache a combo key → confirm order → invalidate runs the fallback path → combo key still present. Either accept and document explicitly as best-effort (the call site already labels it that way) or enumerate combo keys via the in-memory adapter's iterator interface. Not in the production hot path; acceptable as-is.
- **Referenced locations**:
  - `apps/inventory-microservice/src/app/common/modules/product-stock-common/providers/product-stock-common-cache.service.ts` (line ~209, `invalidateNamedKeys` method)
- **Status**: unresolved

### CODE-001 — Defensive filter `!!item.storageId` is unreachable today

- **Code**: `CODE-001`
- **Classification**: other
- **Description**: The `.filter((item): item is typeof item & { storageId: string } => !!item.storageId)` predicate in `ProductStockOrderConfirmService.execute` is unreachable today — every constructed item has `storageId: INVENTORY_DEFAULT_STORAGE`. Recorded as informational and not removed: the type narrowing it produces (`item.storageId: string`) is what lets the next `.map()` emit a `{ storageId: string }` shape that matches `IProductStockCommonCacheInvalidateItem`. A simpler `.map(...)` without the guard would surface a `string | undefined` type error. The filter exists for a forward-looking state where some ledger writes may target a NULL storage column.
- **Referenced locations**:
  - `apps/inventory-microservice/src/app/api/product-stock/providers/product-stock-order-confirm.service.ts` (line ~117, the `.filter((item): item is ...)` predicate)
- **Status**: unresolved
