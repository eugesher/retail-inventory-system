# Simplification Carryover

This document is the **only shared state** across clean-context simplification phases. Each phase reads it on entry and updates it on exit. Treat all content here as load-bearing; do not delete entries from prior phases.

## How to use this document
- On phase entry: read every section in full.
- "Established patterns" is normative — apply listed conventions consistently in your phase.
- "Files modified" tells you what has already been touched; do not re-simplify out of scope.
- "Open / deferred" is a working triage list; consult it for items that may be in your phase's scope.
- On phase exit: update the relevant sections per your task file's Procedure step 5. Do not remove prior entries.

## Established patterns
- **Comments earn their place by carrying WHY**, not WHAT. In particular: leading-class narrations that paraphrase the class name and its type signature ("Framework-free domain entity. Identity is by `id`...") are deleted; comments that carry a non-obvious invariant (e.g. "props must be JSON-stable (no Date / Map / cycles)") are kept and tightened. Phase-01 trimmed these in `libs/ddd/*.ts`.
- **Task-ID / migration-pass references in comments are rot** — `task-04`, `task-09 brief`, `removed in task-14`, "ADR-XYZ records the rationale", and similar pointers were deleted in phase-01. ADR catalogue and git history are the durable records; comments referencing transient task numbers age out fast. Pure ADR-number references that explain a current decision (e.g. "kept in sync with `ROUTING_KEYS`" — the dual-registry policy) are kept because they describe a live constraint, not a past pass.
- **Null/undefined check pattern**: prefer `if (x == null) return ...` over `if (x === undefined || x === null) return ...`. This is the only `==` form the codebase reaches for; everywhere else uses `===`. Applied uniformly in `libs/ddd` (entity-base, value-object-base).
- **Public getters with no production callers are dead surface** even when a spec depends on them — rewrite the spec against the still-public API and drop the getter. Verified in phase-01 by grep across `apps/**`, `libs/**`, and `tests/**` before deletion. Pattern: removing the `domainEvents` getter on `AggregateRoot` collapsed two redundant tests into one without losing meaningful coverage.
- **Hard-constraint frozen surface trumps agent simplification suggestions.** The reuse / quality review agents flagged `BaseTypeormRepository.find/save/softDelete` as potentially dead and `BaseEntity.id` as not-yet-`readonly`; both were correctly left untouched because the phase brief explicitly freezes that public surface. When the simplify review and a hard constraint disagree, the hard constraint wins — note the deferral but make no change.
- **CLAUDE.md-documented public surface behaves like a frozen surface in practice.** Phase-02 found five symbols with zero in-repo callers (`RabbitmqClientFactory`, `CacheHelper`, `@Cacheable`, `MetricsModule`, `TraceContextInterceptor`, and the `ICorrelationPayload` re-export in `libs/observability`) that would normally be deletion candidates per the phase-01 "dead surface" rule. Each is explicitly named in CLAUDE.md as part of its lib's exported surface; the brief makes `CLAUDE.md` out of scope, so deleting them would leave the doc inconsistent. Resolution: keep the symbol, trim any rot in its comment, and record the deletion candidacy under "Open / deferred" so a future pass that updates the docs can close the loop.
- **Placeholder modules / classes are load-bearing imports, not dead code.** `MetricsModule` and `TraceContextInterceptor` look like dead Nest providers but they exist so app modules can already declare the import surface without churning when the real implementation lands. Trim the task-ID rot inside the comment, but keep the placeholder itself.
- **Audit-finding identifiers in comments are preserved as breadcrumbs.** Inline references like `CACHE-010`, `CACHE-011`, `TEST-002`, `TEST-003` follow the same preservation rule as the explicit `AUDIT-2026-05-08` / `AUDIT-2026-05-20` tags called out in the phase brief — they tie a specific mechanism to a specific audit finding, and the link is load-bearing for anyone tracing closure status across `docs/audits/*` and CLAUDE.md's "operational notes" block.
- **Hypothetical future-state postscripts in comments are rot.** Sentences like "The gateway can proxy a `GET /health/notification` to this pattern later" describe code that doesn't exist; they read as planning notes that age into hallucinations. Distinguish from current-state WHY ("RMQ-only service" — true today, structural, explains why `@MessagePattern` is here instead of `@Get`) which is preserved. Pattern verified in `health.controller.ts`.
- **Placeholder adapters keep an ADR-anchored one-line WHY, not a TODO block.** Phase-02 established that placeholder *modules* are load-bearing imports. Phase-03 extends this to placeholder *adapters* (`EmailNotifierAdapter`, `WebhookNotifierAdapter`) — the throwing `send()` is the body; the comment carries the *non-code-visible* WHY ("dependency deliberately not in `package.json`" or "DI slot stays visible alongside the sibling adapter"). The "TODO(post-migration): wire X" form is removed because the throwing body already signals "not implemented" and the wiring sequence belongs in the ADR / a PR description, not in code comments.
- **`*.module.ts` per-module wiring files: keep DI-rebind seam WHY when an ADR documents a rejected alternative.** `notifications.module.ts` retains the "`useExisting` rebind to swap to email/webhook" comment because ADR-011 §3 explicitly rejected the alternative `NotifierService with feature flags` shape — a reviewer seeing `{ provide: NOTIFIER, useExisting: LogNotifierAdapter }` cannot infer from the providers array alone that the feature-flag concrete was considered and rejected. Without a rejected alternative on file, the wiring is self-documenting and the comment is rot. Applied at the per-module `*.module.ts` granularity.
- **Use-case and consumer files were already comment-clean.** Both `SendOrderNotificationUseCase` / `SendLowStockAlertUseCase` and both `OrderEventsConsumer` / `InventoryEventsConsumer` had zero comments at baseline — the class names, decorators (`@EventPattern(ROUTING_KEYS.X)`), and method shapes carry every piece of information a comment would. Treat this as the target shape for the use-case and consumer layers in phases 04–06: if you find yourself writing a comment over a `@MessagePattern` / `@EventPattern` handler that just names the wire pattern, delete it — `ROUTING_KEYS.X` is the doc. Phase-04 confirmed: `StockController` and all four TypeORM entity files matched this target shape with zero edits.
- **Audit-closure breadcrumb blocks compress aggressively when the mechanism lives in an ADR.** Phase-04 trimmed a 12-line "CACHE-001/004 closed by ADR-021 (single-flight in `getOrLoad`, ±10% jitter inside `set`). CACHE-003/009 closed by ADR-022 (schema-version segment + opt-in tenant segment in the key shape). CACHE-005 closed by the `available` flag returned from `get`: when the read fails, `getOrLoad` short-circuits the write-back path so a single Redis-down request produces exactly one warn instead of three." down to a 4-line list. The audit-finding identifiers (`CACHE-001`, `CACHE-004`, `CACHE-005`, etc.) are the load-bearing breadcrumb per the established pattern; the per-closure mechanism narration is rot because the ADR carries the same content with more authority. Keep the **closure mapping** (`X closed by ADR-Y`), drop the **mechanism restatement**.
- **Method-level intentional-no-ops keep a one-line "Intentional X, not Y" framing.** Phase-03 established this for placeholder *adapters* (`EmailNotifierAdapter` body throws). Phase-04 extends it to placeholder *methods*: `StockRabbitmqPublisher.publishStockReserved` returns void with no cross-service consumer, and `StockTypeormRepository.findBySku` returns null because the schema has no SKU column. Both kept a single line of the shape "Intentional null/no-op rather than `not implemented` throw — `<one-clause why>`" because the throwing alternative would force every call site to guard. Drop the future-state narration about why the surface exists at all (that's covered by the port file and/or the ADR).
- **CLAUDE.md-documented external surface absolute-pinning vs internal-surface deletion candidates.** Phase-02 established the "CLAUDE.md-documented public surface behaves like a frozen surface" rule for `libs/**`. Phase-04 found that the analogous rule for `apps/**` is the **audit-document referenced canonical surface**: `infrastructure/cache/spec/stock.cache.spec.ts`'s 9-line canonical-reference header is named in DOCS-001's `informational-no-action` verdict in the 2026-05-20 audit as the canonical convention reference for inventory unit-test layout. Trimming it would invalidate the audit's verdict (the verdict is "we don't need to write the convention in CLAUDE.md because the spec header is the canonical reference"). Distinct rule from the CLAUDE.md case: the cost of removal is not doc-inconsistency, it's audit-record invalidation.
- **Inline interface-method comments are trim candidates by default.** Phase-05 found that `order.repository.port.ts` carried five inline `// Returns X. Used by Y.` comments above method signatures (`findConfirmableOrder`, `customerExists`, `findExistingProductIds`, the `OrderConfirmableProduct` alias, …) — each one paraphrased the method name and parameter types. The interface name + method signature already encode the surface; the inline comments are WHAT-paraphrase. Pattern: delete inline interface-method comments unless they carry a non-obvious WHY (return-shape constraint, performance trade-off, hidden invariant). When a single method *does* carry a non-obvious WHY (e.g. `findOrderResponse` returning joined reference-table fields verbatim because e2e snapshots assert on them), lift it to the file-level header rather than letting it sit inline — file-level WHYs survive signature changes, inline ones decay into stale pointers. Distinct rule from the class-narrating-header pattern (which targets pre-class blocks); this one targets per-method comments inside the interface body.
- **Migration-history spec headers are rot.** Phase-05 deleted the 6-line header on `order.model.spec.ts` that traced the migration from the legacy `OrderConfirmDomain` class ("Migrated from `app/api/order/domain/spec/order-confirm.domain.spec.ts` — the legacy ... folded into `Order.applyInventoryConfirmation`. The migration is durable in git history; the spec file enforces the assertion shape with its actual `expect(...)` calls; the comment is a planning note that aged into a tour-of-the-past. Distinct from phase-04's `domain/spec/` headers (which were comment-clean at baseline): the migration-history spec header is what *replaces* the class-narrating spec header pattern as the next-most-common spec-file rot.
- **Cross-service contract-test claims belong on the adapter method, not at file-level.** Phase-05 relocated the "TypeScript compile is the cross-service contract test" claim (ADR-013 §7) from a method-body comment on `inventory-confirm.rabbitmq.adapter.ts` to a one-line method header on `reserveOrderStock`. Same principle as the interface-method pattern above: hidden-invariant WHYs live where they apply; class-level WHYs go to the class header.
- **Test-coverage-only production surfaces keep an "exists to exercise X" one-line WHY anchored on the test path they enable.** Phase-06 reframed `auth-admin.controller.ts`'s smoke-endpoint comment — `GET /auth/admin/ping` only exists so the e2e suite can exercise the `RolesGuard`'s 403 path (the retail/inventory routes only require `CUSTOMER-or-ADMIN`, leaving the admin-vs-customer rejection otherwise unexercised). Pattern: keep the route + a one-line "exists to cover X — leaving Y otherwise unexercised" WHY + an ADR anchor.
- **Env-variable names in port comments are load-bearing pointers, kept as one-line WHYs when the env is the source-of-truth for a value the method returns.** Phase-06 trimmed `token.port.ts`'s `accessTokenExpiresInSeconds` comment from a 1-line WHAT-paraphrase that opened "Seconds-to-expiry of the access token, …" (the method name already encodes that) to a one-line `// Sourced from JWT_ACCESS_EXPIRES_IN.`. Distinct from the ADR-anchor pattern: env names are runtime-config pointers, not architectural-decision pointers, but they survive the trim under the same "non-obvious + load-bearing" rule.
- **Sibling-file cross-reference compression: when two files document the same mechanism with the same audit-ID, the file *closest to the mechanism* keeps the full narrative; the *consumer-side* file collapses to a one-line breadcrumb + cross-reference.** Phase-07 applied this to the TEST-002 install hook: `test/jest.setup.ts` (where the install happens) kept a 6-line description of the BEFORE-import ordering invariant + the rejected `beforeAll` alternative + the globalThis side-channel + per-Jest-VM-context isolation; `test/system-api.e2e-spec.ts` (which only reads the array back) collapsed to `// Memory-backed Pino capture (TEST-002) — install lives in 'jest.setup.ts'; records from all three apps share one array, distinguishable by 'app'.` Distinct from phase-04's "audit-closure breadcrumb blocks compress when the mechanism lives in an ADR" — that pattern was ADR ↔ comment; this pattern is sibling-file ↔ comment, and the same compression rule applies. The cross-reference (`install lives in 'jest.setup.ts'`) is the load-bearing remainder; everything else is duplication that risks the two narratives drifting out of sync.
- **File:line pointers in comments rot fast — keep the audit-ID and the load-bearing field name, drop the coordinate.** Phase-07 dropped `stock.cache.ts:53–56` from the TEST-002 log-assertion comment in `system-api.e2e-spec.ts`. Line numbers decay the moment the target file is edited; the target file *was* edited in phase-04 (which trimmed `stock.cache.ts` by 33 lines), which would have invalidated the pointer if a future reader had taken it literally. The audit-ID `TEST-002` and the field name `cacheHit: true` survive because they're stable references to *what the assertion proves*; the line numbers were a stable reference to *where to look*, which is exactly the kind of pointer that breaks under simplification passes. Distinct from the established "audit-finding identifiers in comments are preserved as breadcrumbs" rule: the *identifier* persists, the *coordinate* does not.
- **Coverage-matrix breadcrumbs (G*, C*, etc.) survive even when undocumented, but concrete *examples* embedded next to them must track current state.** Phase-07 kept all six G1-G5 / C1-C5 tags in `system-api.e2e-spec.ts` even though `grep` across `docs/` finds no published matrix — each tag is 2-3 chars and ties a specific test to a specific cache-aside branch, a cheap and durable mental-map aid. Pattern: keep the breadcrumb identifier; verify any concrete example embedded next to it against the current code/key shape; drop the example if stale or replace with a current form. Distinct from phase-04's "audit-closure breadcrumb blocks" rule (which is about *audit-finding* identifiers tied to `docs/audits/`): coverage-matrix identifiers are author-shorthand without an external source, but they earn the same preservation under the same "cheap + durable + ties to a logical branch" reasoning.
