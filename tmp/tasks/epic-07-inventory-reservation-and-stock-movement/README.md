# Epic 07 — Task Index

Decomposition of `tmp/epics/epic-07-inventory-reservation-and-stock-movement.md` into 13 sequential, self-contained execution tasks. Each task file states its entry-state assumption (what previous tasks — and `epic-04` / `epic-05` — leave on disk), the concrete files to add/modify/delete, the doc deliverable, and the exit criteria. The decomposition mirrors the layout used for `epic-02`, `epic-04`, and `epic-06`, and follows the epic's own 13-row `Carryover Between Tasks` table 1:1.

This epic **extends the existing `stock` module** introduced by `epic-04` — it adds no new bounded context and no new microservice. It does add: two new aggregates (`Reservation`, `StockMovement`) inside `stock/domain/`, one new sibling repository port (`RESERVATION_REPOSITORY`), three new emit methods on the existing `IStockEventsPublisherPort`, four new routing keys + three new RPC patterns, a new `INVENTORY_RESERVATION_GATEWAY` port on the retail side (mirroring `epic-13`'s `INVENTORY_CONFIRM_GATEWAY`), and three new api-gateway endpoints. The difficulty is concentrated in the **no-oversell invariant** (transactional + OCC retry) and the **cross-service reserve/allocate chain**, not in new infrastructure scaffolding — so `eslint-plugin-boundaries` (ADR-017) needs no rule changes.

## Sequence and dependencies

Tasks depend on each other through the `Carryover Between Tasks` table in the epic. Do them in order; do not parallelize. Tasks 01–02 stand up the two new tables + aggregates + repositories (`Reservation` with `@VersionColumn`; `StockMovement` append-only). Task-03 implements the two cart-write use cases (Reserve / Release) with the OCC retry policy and ADR-023 post-commit cache invalidation, and extends the events publisher with the three mandatory emits. Task-04 implements Allocate (Reservation→committed, with a fallback and an expired-reservation rejection). Task-05 implements Cancel Allocation behind an RPC handler stub for `epic-08`. Task-06 closes the `epic-04` deferral by having Receive/Adjust write `StockMovement` rows. Task-07 adds Transfer Stock (two-movement transaction + endpoint). Task-08 wires the cart-side RPCs through the new `INVENTORY_RESERVATION_GATEWAY` port and retires the legacy `inventory.order.confirm` handler. Task-09 adds the movements-audit + ops-release HTTP endpoints. Task-10 bumps the cache key version `v2`→`v3`. Task-11 authors the required concurrent-oversell e2e plus the reserve/release/allocate/audit e2e suites. Task-12 extends `http/inventory.http`. Task-13 closes out with seeds, README/CLAUDE.md, and the architecture-lint append-only fixture.

## Cross-epic coupling notes

- **`epic-04`** is a hard prerequisite. It ships the `stock` module, the `StockLevel` aggregate with the `version` column from day one (so this epic's OCC retrofit is non-destructive — no schema change), the `stock_location` table with the auto-provisioned `default-warehouse`, the `STOCK_REPOSITORY`/`STOCK_CACHE`/`STOCK_EVENTS_PUBLISHER`/`TRANSACTION_PORT` quad, and the `v2` cache key. This epic adds `quantityReserved`/`quantityAllocated` *mutation* (epic-04 shipped the columns at default `0`, written but never moved).
- **`epic-05`** is a hard prerequisite. It ships the cart/order walking skeleton in `retail-microservice` (the Cart aggregate, the `Add to Cart` / `Remove` / `Change Quantity` / `Place Order` use cases) and the api-gateway `/api/cart/*` endpoints. Task-08 changes those use cases' *behavior* (they now call inventory reservation RPCs) without changing their HTTP signatures.
- **`epic-02`** supplies `product_variant` — the cross-service FK target for `reservation.variant_id` and `stock_movement.variant_id` (enforced by convention, not a SQL `REFERENCES` clause; see ADR-019).
- **`epic-01`** supplies the gateway RBAC primitives the task-07/task-09 endpoints gate on (`PermissionsGuard`, `@RequiresPermission()`, `PermissionCodeEnum`). Task-13 seeds the new `inventory:transfer` code into `warehouse-staff`.
- **`epic-08`** is the natural successor: it owns **Commit Sale** (the on-ship physical decrement + the `StockCommitted` event) and **Cancel Order**, which calls task-05's Cancel Allocation RPC. Task-05 ships the RPC handler stub so epic-08's caller has a wire to bind to.
- **`epic-09`** owns **Restock from Return** (the `return`-type StockMovement producer). This epic ships the `return` enum value but no producer.
- **`epic-11`** owns the event-store consumer that subscribes to `inventory.stock-movement.recorded` (the high-volume catch-all this epic produces for every movement insert). This epic produces the event; it does not add the audit binding.
- **`epic-12`** hardens the OCC story (idempotency-key dedupe on Reserve; OCC on Cart) and adds broader concurrency integration tests. This epic ships the retry-then-409 policy and the `@VersionColumn`; epic-12 generalizes them.
- **`epic-14`** owns the **Reservation TTL sweeper** background job (the wall-clock `active`→`expired` flip). This epic ships only the entity, the inline TTL check at allocate-time, and the manual-release endpoint; the `(status, expires_at)` index is created here for that future sweeper.
- **`epic-15`** owns every excluded inventory concern (lot/batch/serial, expiry/FIFO, bin/aisle/shelf, transfer-order documents, in-transit-as-separate-location, multi-location order routing). This epic always reserves/allocates at `default-warehouse`; the schema carries `stockLocationId` everywhere so the routing retrofit is non-destructive.
