# Epic 04 — Task Index

Decomposition of `tmp/epics/epic-04-inventory-stock-level-and-location.md` into 10 sequential, self-contained execution tasks. Each task file states its entry-state assumption (what previous tasks left on disk), the concrete files to add/modify/delete, the doc deliverable, and the exit criteria. The decomposition mirrors the layout used for `epic-01`, `epic-02`, and `epic-03`.

## Sequence and dependencies

Tasks depend on each other through the `Carryover Between Tasks` table in the epic. Do them in order; do not parallelize. Task-01 detonates the old ledger-as-source schema (`product_stock` + `product_stock_action` + `storage` + the inventory-side `product` stub) and parks `StockTypeormRepository` in a throwing-stub state. Tasks 2–3 stand up the replacement tables (`stock_location` + `stock_level`) one at a time, with the default warehouse auto-provisioned at the migration boundary. Task-04 swaps the `StockItem` aggregate for `StockLevel`. Task-05 rewrites the use case layer — `add-stock` and `reserve-stock-for-order` are deleted (Reservation is deferred to `epic-07`), `get-stock` is reshaped into `query-availability`, and two new use cases (`receive-stock`, `adjust-stock`) land. Task-06 bumps the cache key version (ADR-022) from `v1` to `v2` and reshapes the StockCache payload. Task-07 binds the `catalog.variant.created` consumer that auto-initializes `StockLevel = 0` at the default location; the level-initialized routing key + publisher-port method + adapter implementation ship in the same task so the auto-init use case never sits on a raw `ClientProxy` injection (per ADR-008 §"Domain code depends on a publisher port" — see `epic-00/task-10`). Task-08 wires the remaining two emit-side routing keys (`stock.received`, `stock.adjusted`), reshapes `publishStockLow`'s payload, retires `INVENTORY_PRODUCT_STOCK_GET`, and converts `INVENTORY_ORDER_CONFIRM` to a deprecation handler. Task-09 rebuilds the api-gateway `modules/inventory/` from `productId`-keyed to `variantId`-keyed and replaces `http/product.http` with `http/inventory.http`. Task-10 closes out with seed extensions, README/CLAUDE.md updates, and architecture-lint fixtures.

## Cross-epic coupling notes

- **`epic-02`** is a hard prerequisite. The new `stock_level.variant_id` is a cross-service FK to `product_variant.id` (catalog-side); the FK is enforced at the application layer + via the auto-init consumer of `catalog.variant.created`, not as a SQL `REFERENCES` clause (two services share a MySQL schema today, but ADR-019 keeps each microservice's tables self-owned at the schema level so each owns its migrations). Task-07's consumer cannot bind until epic-02 task-03 has registered the `catalog.variant.created` routing key. Task-02 of epic-02 (the `DropInventoryProductTable` migration) is now ordering-sensitive with this epic's task-01: if epic-02 task-08 ran first, this epic's task-01 must skip the `product` drop and log it as a no-op; if it did not, this epic's task-01 owns the drop. Task-01 below documents both branches.
- **`epic-03`** is concurrent but not coupled. Pricing's `Publish Product` hard-fail does not touch inventory. The `catalog.price.changed` / `catalog.price.scheduled` events have no inventory consumer (deliberately — pricing changes do not invalidate stock cache).
- **`epic-07`** is the natural successor. It owns `Reservation` + `StockMovement` + the four operations deferred here (`Allocate`, `Commit Sale`, `Cancel Allocation`, `Restock from Return`). This epic ships the `version` column on `stock_level` from the start so the OCC retrofit in `epic-07` and the hardening in `epic-12` are non-destructive. Task-04 below documents this contract explicitly so a future `epic-07` contributor does not regress the column away thinking it is dead weight.
- **`epic-08` / `epic-09`** own fulfillment + returns; neither is touched here. `Transfer Stock` (multi-warehouse routing) is `epic-15`'s exclusions register — the universal core does not include it.
- **`epic-11`** owns the audit-log consumer. Task-08's three new routing keys are produced for `epic-11` to subscribe; this epic does not add the audit binding.
- **`epic-12`** hardens OCC with retry policy + integration tests. Task-04 below ships the `version` column with `@VersionColumn()` and the per-mutation bump in the aggregate; `epic-12` adds the conflict-retry wrapper.
- **`epic-15`** owns every excluded inventory concern (lot/batch/serial, expiry/FIFO, bin/aisle/shelf, demand forecasting, transfer-order documents, consigned inventory, ABC classification, in-transit-as-separate-location, multi-location order routing). No Exclusions Register entries are added by this epic.
