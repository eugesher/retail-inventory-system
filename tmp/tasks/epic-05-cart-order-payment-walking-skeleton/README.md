# Epic 05 — Task Index

Decomposition of `tmp/epics/epic-05-cart-order-payment-walking-skeleton.md` into 12 sequential, self-contained execution tasks. Each task file states its entry-state assumption (what previous tasks left on disk), the concrete files to add/modify/delete, the doc deliverable, and the exit criteria. The decomposition mirrors the layout used for `epic-01` through `epic-04`.

## Sequence and dependencies

Tasks depend on each other through the `Carryover Between Tasks` table in the epic. Do them in order; do not parallelize. Task-01 detonates the legacy retail schema (`order` + `order_product` + `order_status` + `order_product_status` + `customer`) and parks `OrderTypeormRepository` in a throwing-stub state — the same idiom used in epic-04 task-01. Tasks 02–04 stand up the replacement tables one aggregate at a time: `cart` + `cart_line` (02), then `order` + `order_line` + polymorphic `address` (03), then `payment` plus the `PAYMENT_GATEWAY` port + `FakePaymentGatewayAdapter` (04). Tasks 05–08 land the use case layer in topological order — cart mutators first (05), then Place Order with its cross-service catalog RPCs and inline `PAYMENT_GATEWAY.authorize` call (06), then Capture Payment (07), then the two reads `GetOrder` / `ListMyOrders` (08). Task-09 reshapes the api-gateway side: a new `modules/cart/` and a rename of `modules/retail/` → `modules/orders/`, both hexagonal. Task-10 ships the Kulala HTTP files (`http/cart.http` new, `http/order.http` rewritten). Task-11 re-points the notification microservice's `order-events.consumer.ts` from the retired `retail.order.created` to the new `retail.order.placed` so the cross-cluster chain stays intact. Task-12 closes out — seed extension, three new e2e specs, README/CLAUDE.md rewrites, architecture-lint fixtures for the new `cart/` and `payment-gateway/` element types.

## Cross-epic coupling notes

- **`epic-01`** is a hard prerequisite. Customer identity lives only in the api-gateway's auth module (the `default-b` decision); this epic deletes the retail-side `customer` table accordingly. Permission codes `customer:own-orders:read` and `order:capture` are seeded by task-12 against the role schema epic-01 established (customer role auto-gets `customer:*`; admin + order-support get `order:capture`).
- **`epic-02`** is a hard prerequisite. The Place Order use case (task-06) reads variant metadata (`sku`, `name`) via the catalog microservice's variant RPC (registered by epic-02 task-03). The retail-microservice imports `MicroserviceClientCatalogModule` to obtain the catalog `ClientProxy`.
- **`epic-03`** is a hard prerequisite. The cart-line `unitPriceSnapshot` at Add to Cart (task-05) and the order-line `unitPriceMinor`/`taxAmountMinor` at Place Order (task-06) both call `catalog.price.select-applicable` (registered by epic-03). The "snapshot is the contract" principle is documented in `04-order-line-snapshot-and-cross-service-lookup.md`.
- **`epic-04`** is a soft prerequisite — the inventory side's `query-availability` is not invoked by this epic's walking skeleton (Add to Cart has no stock check until `epic-07` introduces Reservation). Co-existing is enough; no schema or routing-key coupling.
- **`epic-07`** is the natural successor. It owns `Reservation` (cart-time stock hold), `Allocate Stock at place-time`, the `StockMovement` ledger row produced from Receive/Adjust, and the `OrderLine.status='allocated'` real allocation that this epic ships as a forward-compatible sentinel. The `version` column on Cart and Order ships now so the OCC retrofit in `epic-12` is non-destructive (this epic accepts but does not enforce optimistic locking).
- **`epic-08`** owns ship + fulfillment + the ship-triggered automatic Payment Capture; this epic ships only the explicit-capture endpoint to be coherent today. Both `Order.status` and `Order.fulfillmentStatus` carry forward to that epic unchanged.
- **`epic-09`** owns returns + refunds + `RETAIL_ORDER_CANCELLED`'s eventual producer/consumer.
- **`epic-10`** owns the rest of the notification re-pointing for the cluster reshape (`retail.payment.captured` / `retail.cart.created` / etc. consumers). This epic keeps the bare minimum re-point (task-11 → `retail.order.placed`) so the cluster's known-good notification chain does not break across the schema rewrite.
- **`epic-11`** owns the audit-log consumer. The state-transition events emitted by tasks 05 / 06 / 07 (cart + order + payment) feed the audit consumer when it lands. This epic does not add the audit binding; the `AUDIT_LOG_PUBLISHER` port (no-op default from `epic-01`) is invoked at each transition so the call sites are already correct.
- **`epic-12`** hardens idempotency-key dedupe + OCC retry. This epic accepts the `Idempotency-Key` header on Place Order and Capture Payment but does not deduplicate yet. Task-06 and task-07 mark the header-forwarding path as a TODO referencing `epic-12`.
- **`epic-15`** owns every excluded payment concern (real gateway integrations; BNPL; B2B quote/PO; fraud screening; tax computation; shipping rate calculation). The `FakePaymentGatewayAdapter` is replaced behind the `PAYMENT_GATEWAY` port without touching application code.
